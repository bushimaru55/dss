## 3. 004 metadata と 006 logical path の関係

### 3.1 役割の違い（比較整理）

| 観点 | 004 `AnalysisMetadata`（概念・API） | 006 logical path（論理参照） |
|------|-------------------------------------|--------------------------------|
| 主な役割 | **分析意味の器**。dimensions / measures / time_axis 等を **JSON として保持** | **論理フィールドを指す表記**（例: `affected_fields`、`required_fields` 想定）。**006** が I/O・整合の語彙を担う |
| MVP での観測 | API・DB 上の **実在フィールド**として返る（id・配列の中身は **観測**） | **厳密な path 命名規約の正本は未固定**（Phase 4 目標が SPEC に散在） |
| 013 との関係 | **stub evidence の主たる入力**（trace） | **現時点では 013 に載せない**（§19.6）。将来 `required_fields` 等で接続しうる |

**結論（断定ではない）**: 004 は **「何がメタとして存在するか」**、006 logical path は **「論理上どのフィールドを指すか」** の層。**同一文字列を機械的に写すだけでは足りない**場合があり、**対応表（mapping）は別イテレーションで明示**するのが安全。

### 3.2 フィールド単位の整理（比較）

以下、**safe observation**（観測としてそのまま説明に使いやすい）と **logical path 側**（命名・正本がまだ揃っていない）を分ける。

| 項目 | safe observation（現行で言いやすいこと） | logical path / 未固定 |
|------|------------------------------------------|------------------------|
| **`metadata_id`** | 004 の主キー。**システム ID**。013 evidence に載せうる | path ではないが、**他リソースとの紐づけキー**として安定 |
| **`dataset_id`** | 003/004 接続の **FK**。観測として載せうる | 論理列を指すものではない |
| **`dimensions[]` / `measures[]`** | 要素の **id / key / name** を **列挙・写し**（Pattern B）。**意味確定ではない** | **`dimensions.xxx` / `measures.xxx` 形式の統一 path**は **006 側の確定待ち** |
| **`time_axis`** | **有無**や JSON の **存在**は観測。**中身の解釈は MVP で固定しない** | **time 系の logical path**（月次・期間等）との **1:1 対応は未固定** |

**path 命名規約が未固定なため保留になるもの**: **006 準拠のドット記法の正本**、`grain` を path のどこに置くか、**003 `column_slots` から 004 要素への昇格規則**の機械化、など。

### 3.3 004 metadata ↔ 006 logical path の最小対応表ドラフト

本節は **ドラフト** である。**対応表の機械可読フォーマット・path 命名の正式決定は行わない**。目的は、**各行（観測要素）について「何が今言えるか／何が保留か」を一表に並べる**ことにある。

#### 3.3.1 対応表

各行は **004 に現れる観測単位**に対応する。**006 logical path** は **将来の論理参照**の列に置き、**現時点では正本を仮定しない**。

| 観測単位（行） | 004 側の観測要素 | 現時点で safe observation として扱えるもの | 将来 006 path と対応づけうる論理対象（候補・未確定） | 現時点で未固定な点 | 013 stub evidence で使ってよい範囲 | `required_fields` 導入時に追加検討が必要な点 |
|----------------|------------------|--------------------------------------------|------------------------------------------------------|---------------------|-------------------------------------|------------------------------------------------|
| **`metadata_id`** | 004 `AnalysisMetadata` の **主キー**（当該メタ行の識別子） | **存在**・**同一分析メタ実体の参照**として説明できること。他 API / evidence から **トレース上で指し示す**こと | **メタ実体を束ねる外部参照**（path の末端セグメントそのものではない想定が自然）。006 側で「どのメタを対象にするか」を指す **論理アンカー**になりうる | **版・複製・再生成**時に ID がどう振られるかの運用。**メタ粒度**（分割・マージ）と ID の関係 | **§19.6 に沿う説明用トレース**として **ID の写し・参照**は可。**path 正本の主張はしない** | メタ全体を **必須対象**に含めるモデルか、**部分 path のみ**か。**必須性のスコープ**（メタ vs 列） |
| **`dataset_id`** | 003/004 接続の **データセット FK**（どの入力データを対象にしているか） | **どのデータセットを分析対象としているか**の観測。**003 との整合**を説明する材料 | **データセットスコープ**の指定（**列を指す logical path とは別軸**）。将来 **dataset 境界**と path を **結合する規則**の入力になりうる | **マルチテーブル・スナップショット・差し替え**時の ID の意味。**dataset と column path の優先順位** | **観測としての ID 写し**は可。**「この path が dataset 内で有効」**の主張までは stub では **断定しない** | **`required_fields` の解釈を dataset で閉じるか**、**跨ぎ**を許すか。**スコープ結合**の明示 |
| **`dimensions[]`** | **次元の列挙**（要素ごとに id / key / name 等を含みうる JSON 配列） | **列挙の有無**、**要素数**、**各要素に付与されている識別子・表示名の写し**（**意味の確定・階層の確定までは観測に留める**） | **論理次元軸**（例: `dimensions.<stable_id>` 形式の **候補**。**正式表記は 006 未確定**）。**階層**がある場合は **複数セグメントや別表現**の可能性 | **要素の正本キー**（id vs key vs name）。**並び順の意味**。**親子・ロールアップ**の表現が 004 JSON に載るか | **Pattern B に沿う観測トレース**（**dimensions の写し**）。**「この文字列が canonical path」**とは言わない | **配列要素 1 件 ↔ path セグメント**の対応規則。**省略・デフォルト**。**複数ディメンションの合成 path** |
| **`measures[]`** | **指標の列挙**（同上、JSON 配列） | **列挙の有無**、**要素の写し**。**集計対象として何が列挙されているか**の観測 | **論理指標**（例: `measures.<stable_id>` **候補**）。**measure と column_slot / 物理列**の関係は **別レイヤの対応表**になりうる | **measure 要素の正本キー**。**同名異義**。**計算 measure** の表現 | **観測写し**は可。**path 正本の昇格はしない** | **必須指標の path 化**。**dimension との組み合わせ**（slice）を `required_fields` にどう載せるか |
| **`time_axis`** | **時間軸メタ**（JSON。**構造・キーは実装・データ依存**） | **時間軸情報の有無**、**オブジェクトとしての存在**、**主要フィールドの写し**（**解釈の単一化は MVP で強制しない**） | **時間に関する論理参照**（**time dimension**、**period フィールド**、**date hierarchy** 等、**複数の表現可能性**）。**単一路 path への 1:1 対応は仮定しない** | **単一の canonical path に落とすか**。**time を dimension と同一語彙で扱うか別軸か**。**粒度（grain）**をここに読むか path 側か | **有無・写しのトレース**は可。**「time = この path 一択」**は stub では **言わない** | **time を `required_fields` のどこに置くか**。**grain との二重定義防止**。**期間型・スナップショット日**との関係 |

**表の読み方（再掲）**: **左から右へ「観測 →（将来）論理参照」へ橋をかける候補**であり、**右端の列は導入時の論点メモ**である。**セル内容は確定仕様ではない**。

#### 3.3.2 dimensions / measures の識別子規約メモ

**本項は正式規約を確定しない**。MVP の JSON および Pattern B の写し方から **現状こう読めること**と、**将来揃えると対応表が書きやすくなる点**を分ける。

| 識別子系 | 表示向けとして読みやすいもの | 内部参照向けとして使いやすいもの | 将来 path 対応の「足場」になりやすいもの（候補） | いま保留になりやすいこと |
|----------|------------------------------|----------------------------------|------------------------------------------------|--------------------------|
| **`name`** | **人間向けラベル**。ダッシュボード表示・説明文にそのまま使いやすい | **一意性が保証されない**場合がある。**ロケール・表記ゆれ** | **path の唯一の根拠**にはしにくい（**表示と論理 ID の分離**が安全） | **同名異義**のときの区別方法 |
| **`key`** | **短いスラッグ**として表示に使うことはある | **実装・JSON 間の安定参照**として使われうる（**運用で揃っていれば**） | **006 のセグメント名**に寄せる設計にしやすい（**ただし正本は未固定**） | **key の付与規則**がデータソース依存のときの一貫性 |
| **`id`** | **そのまま UI に出す**より **内部向け**に見えることが多い | **システム発番・永続 ID**として **参照の軸**になりやすい | **`measures.<id>` / `dimensions.<id>`** のような **安定セグメント**の候補（**006 が採用するかは別問題**） | **id のスコープ**（グローバル / dataset 内 / メタ版内） |

**比較上の整理**: **表示**は `name`、**機械参照**は `id` または運用で固定された `key`、**将来 path** は **「どれを canonical とみなすか」**の合意が先に必要。**三者が同一文字列である必要はない**（同一だと楽だが、仕様上は未要求）。

**一段進めた整理**（**canonical key の仮置き**、**path / requirement の仮軸**）: **§3.6**。

#### 3.3.3 time_axis の扱い

- **現時点の中心**: **有無**および **JSON として何が観測されているか**（§19.6 の **observation trace** の範囲）。**単一路の logical path と同一視しない**。
- **将来ありうる分解**（**併存しうる**ため、**このメモでどれか一つに決めない**）:
  - **time dimension**（他の dimensions と並べる語彙）
  - **period / as-of フィールド**（会計期間・スナップショット日）
  - **date hierarchy**（年→四半期→月 等。**階層 path** は **複数セグメント**になりうる）
- **対応表との関係**: **time_axis 行**は **「論理時間の置き場」が複数候補ある**ことを示す。**006 で path を一本化する**のは **後続の合意**であり、**stub evidence で path 正本を先取りしない**。

### 3.4 006 の最小 path 語彙ドラフト

**本節はドラフト**である。**006 の path 命名規約の正式決定・正本化は行わない**。目的は、§3.3.1 で並べた **004 観測単位**に対し、**006 側で「論理参照の語彙」として最低限どのような束が要りそうか**を **候補レベルで言語化**することにある。

**§3.3.1 対応表との接続（三者の関係）**:

| レイヤ | 役割（本メモでの位置づけ） |
|--------|----------------------------|
| **004** | **観測の受け口**。dimensions / measures / time_axis 等は **JSON 観測写し**（§3.3）。**意味確定・path 正本の主張はしない** |
| **006** | **logical path の候補整理**（本節）。**将来の I/O・整合・必須フィールド列挙**に耐える **語彙のたたき台** |
| **013** | **evidence は observation trace**（§19.6）。**本節の path 候補を evidence に載せても、正本化したことにはならない** |

**対象ごとの整理（概念分類・ドラフト）**:

| 対象 | path 化の対象になりやすいか（傾向） | path というよりスコープ / 属性 / 補助に近いか | 先に固定しないと path 化が詰まりやすいこと（例） |
|------|--------------------------------------|------------------------------------------------|--------------------------------------------------|
| **dimensions** | **個別次元**は **「名前付きの論理軸」**として **セグメント化しやすい**傾向 | **配列順・親子**などは **単一路 path に落ちない**場合がある | **canonical key**（id / key / name のどれを軸にするか）、**階層表現**を path に含めるか |
| **measures** | **個別指標**は **論理列に近く**、**`measures.<key>` 系の候補**と **対応表を書きやすい**傾向 | **計算 measure・派生**は **path 1 本**に収まらない可能性 | **measure と物理列 / slot**の対応、**同名異義**の解消規則 |
| **time_axis** | **単一の「時間フィールド」**に還元できる場合は path 候補になりうるが **必ずしも単一ではない** | **期間型・as-of・階層**は **属性束・複数セグメント**の方が自然な場面がある | **time を dimension と同一語彙にするか別軸か**、**1:1 の単一路**を仮定するか |
| **grain** | **「集計の解像度」**を **1 セグメント**で表す案もあるが **単位・暦**の解釈が絡む | **004 の観測**と **006 の必須性**の **どちらに寄せるか**で **path vs メタ属性**が分岐しうる | **API 正本位置（§5）未固定**。**time_axis との二重定義**の回避 |
| **dataset scope** | **データセットを「列 namespace」**とみなすなら **path 接頭辞**に見立てうる | **多くの設計では FK・スコープ引数**として **path 本体から分離**しやすい | **path 解釈を dataset で閉じるか**。**跨ぎ**の有無。**metadata との優先順位** |

**読み方**: **dimensions / measures は path 候補になりやすい**一方、**time_axis・grain・dataset（およびメタ ID）は「同じノリでドット記法の列だけに並べる」と **すぐ破綻しうる**（後述 §3.4.1）。

#### 3.4.1 path 化しやすい概念 / しにくい概念

**path にしやすい候補（傾向）**

- **measures の個別対象**: **004 で要素として列挙されている**単位が、**論理指標 1 件 ↔ path の末端（または短いセグメント列）**として **対応表を引きやすい**。**先に必要なのは**、要素キーの **スコープ内一意性**の仮説（§3.3.2）。
- **dimensions の個別対象**: **同上**。**論理次元 1 件**として **識別子が安定**していれば、**path の足場**になりやすい。**階層がある場合**は **単一路**にしない設計もありうる。

**path にしにくい / 追加前提が必要な候補**

| 候補 | なぜそう言えるか（要約） |
|------|---------------------------|
| **`time_axis`** | **単一路 path と 1:1 でない**場合がある（§3.3.3）。**time dimension / period / hierarchy** など **表現が分岐**しうるため、**先に「何を論理フィールドと呼ぶか」の合意**がないと **文字列だけでは結合できない**。 |
| **`grain`** | **列ではなく解像度・単位のセマンティクス**に近く、**004 の観測**と **006 の必須性**の **どちらを正とするか**が未固定（§5）。**path の 1 セグメントに載せるか**は **単純化しすぎるリスク**がある。 |
| **`dataset_id`** | **列・論理軸の識別子ではなく**、**入力データの境界**。**path の外でスコープとして付与する**方が **解釈がぶれにくい**場面が多い（**ただし接頭辞案も比較対象**、§3.4.2）。 |
| **`metadata_id`** | **分析メタ実体の ID**であり、**論理フィールド path の一部**とは **通常別物**。**API やクエリのスコープ**として **別引数**に置くモデルと **整合**しやすい（§3.4.3）。 |

#### 3.4.2 最小語彙候補の比較

以下は **記法のドラフト比較**である。**いずれも正式採用ではない**。**並び・命名は例示に過ぎない**。

| 候補（例示） | 役割イメージ | path としての読みやすさ | §3.3.1・004 との橋の作りやすさ | 留意（ドラフト） |
|--------------|--------------|-------------------------|----------------------------------|-------------------|
| **`dimensions.<key>`** | **論理次元の 1 要素**を指す **セグメント列の候補** | **ドット記法で説明しやすい** | **dimensions[] 要素の canonical key** と **対応表が引きやすい** | **`<key>` の正本**。**階層を `dimensions.region.city` のように伸ばすか**は未固定 |
| **`measures.<key>`** | **論理指標 1 件** | **同上** | **measures[] 要素と対応しやすい** | **計算・派生**をどう表すか |
| **`time.<key>`** | **時間に関する論理参照**の **束**（**time dimension・単一フィールド**想定のとき） | **time を dimension と分離**する語彙として **並べやすい** | **time_axis JSON** から **単一 key に落とせる場合**に橋がかかる | **落とせない場合**は **複数セグメント・別表現**が必要。**`time` と `dimensions` の重複**に注意 |
| **`grain`（単独または `time.grain` 等）** | **集計粒度** | **1 セグメントで語れる**ときは **短い** | **004 の time_axis / grain 観測**との **対応表が要る** | **正本位置未固定**。**time と二重定義**しうる |
| **`dataset.<key>`** | **データセットを path 接頭辞に見立てる**案 | **「この文字列以下が有効」**と読める | **dataset_id を文字列化**すれば **対応は単純**に見える | **path 本体と直交するスコープ**の方が **自然な API** も多い。**key は id か別名か** |

**比較の結論（断定しない）**: **列に近い dimensions / measures** は **語彙の核**になりやすい。**time・grain・dataset** は **同じドット記法に無理に揃える**より、**スコープ・属性・複数セグメント**として **分けて設計**した方が **矛盾が少ない**場面がありうる。

#### 3.4.3 dataset_id / metadata_id を path と呼ぶか

**現時点の整理（ドラフト）**:

- **`metadata_id`**: **path の末端が指す「論理フィールド」**とは **別カテゴリの識別子**として扱うのが **素直**なことが多い。**「`metadata.<id>` を path と呼ぶ」**案は **比較対象**にはなるが、**006 の logical path が主に担うべきもの**（列・軸の参照）と **混線**しやすい。**別パラメータ・別ヘッダ・リクエストスコープ**として **path 文字列と合成する**運用の方が **説明しやすい**場合がある。
- **`dataset_id`**: **列 path の親**ではなく **データ境界**。**`dataset.<id>` を path の先頭に付ける**案は **namespace として一貫**しうる一方、**必ずしも 006 の「logical path」定義に含める必要はない**。**「path は dataset 内で解釈」**を **規則として外に出す**案と **トレードオフ**になる。

**本項の位置づけ**: **「path という語を何に使うか」**（**文字列 1 本の中身**か、**スコープ付き論理参照の総称**か）を **早めにすり合わないと**、`dataset` / `metadata` の扱いで **実装が分岐**しうる。**ここでは結論を確定しない**。

### 3.5 006 path 文字列の意味範囲

§3.3・§3.4 までで、**004 は観測**、**006 は logical path の候補**、**013 は trace**、**dimensions / measures は path 化しやすく**、**time・grain・dataset・metadata は同じノリで path 化しにくい**、ところまで整理した。

本節では **一段内側**として、**006 の「path 文字列」という語が、そもそも何を指すか**の **意味範囲**を **比較**する。**正式な定義・命名規約の確定は行わない**。**path を何でも入る箱にしない**ことと、**`required_fields` 議論の前に意味範囲を揃える**ことを **目的**とする。

**主題（比較の軸）**: path は **「列・軸を指す論理参照」に限定するのか**、それとも **粒度・期間・スコープまで含む広い参照記法**とみなすのか。

#### 3.5.1 狭義案 / 中間案 / 広義案の比較

| 案 | path に **含める**（イメージ） | **分かりやすい**点 | **危ない**点 / 負担 | **004 / 006 / 013 / `required_fields` 検討**との相性（傾向） |
|----|-------------------------------|--------------------|---------------------|----------------------------------------------------------------|
| **A. 狭義案** | **列・軸の論理参照**に限定。例: **`dimensions.*` / `measures.*`**（§3.4 の核に近い） | **「何の列・軸を指すか」**が **一文で説明しやすい**。**機械的整合**（同一キーか）も **検証しやすい** | **time・grain・スコープ**を path の外に出すと、**別フィールド・別リスト**との **合成規則**が **別途必要** | **004** の dimensions/measures 観測と **橋をかけやすい**。**006** は **語彙を絞り込める**。**013** の trace と **混線しにくい**（**path 正本の主張と observation の差**が **説明しやすい**）。**`required_fields`** は **「列・軸必須」**に **寄せやすい** |
| **B. 中間案** | **主に列・軸**だが、**条件付きで time 系**も path に **含みうる**（例: **`time.<key>`** が **単一キーに落ちる**場合のみ、等） | **時間軸**を **論理フィールド**として **必須化したい**場面で **一列に並べられる**。**狭義と広義の中間**として **柔軟**に見える | **「いつ time を path に入れるか」**の **境界**が **曖昧**になりやすい。**004 の time_axis** が **単一路に還元できない**とき **破綻**しうる（§3.3.3） | **004↔006 の対応表**に **条件分岐**が増えやすい。**013** は **time を path に載せた説明**と **観測トレース**の **差分説明**が **やや重くなる**。**`required_fields`** は **列必須に加え「時間論理の必須」**を **載せられる**が、**狭義より解釈コストが上がる** |
| **C. 広義案** | **列・軸**に加え、**粒度・期間・スコープ**まで **同一の path 記法**に **含める**（**dataset / metadata / grain を文字列に載せうる**） | **「この文字列 1 本で文脈が揃う」**ように **見える**。**クライアントが 1 フィールドだけ渡す**設計に **寄せやすい** | **意味範囲が広いほど**、**path 文字列が正本**に見えるリスク。**§19.6**（**evidence は trace**）や **`required_fields`（論理参照）**と **同じ文字列を混同**しやすい。**検証・デバッグが難しくなる**場合がある | **004** の **観測 JSON 全体**を **path に押し込めたくなる誘惑**と **衝突**しやすい。**013** の **stub evidence** と **論理 path** の **境界がぼやける**。**`required_fields`** が **列必須・粒度必須・期間必須・スコープ**を **同一リストに混在**させ、**優先順位・解釈**が **曖昧**になりうる |

**比較のメモ（断定しない）**: **広義**は **表現力**は高く見えるが、**責務の分離**（観測 / 論理 / スコープ）と **相性が悪くなりやすい**。**狭義**は **足りないものを別枠で持つ**ことが **前提**になる。

#### 3.5.2 現時点で安全な仮置き

**以下は正式決定ではなく、議論の仮置き**である。**後から変えうる**前提で書く。

**仮置き（現時点で無理が少ない方向）**: **006 の path 文字列**は **まず A（狭義）に寄せて考える**、すなわち **「論理列・論理軸（dimensions / measures の個別対象）を指す記法」**として **意味範囲を狭く保つ**、という整理が **比較的安全**に見える。

**理由（観点だけ列挙）**:

- **MVP 後続の整合**: **006 が担うもの**を **「論理フィールド参照」**に **寄せたまま**、**004 の観測**との **対応表**を **増やしすぎず**に済みやすい。
- **013 trace との混線**: **path 文字列**が **観測トレースの写し**と **同一視されにくい**（**§19.6** の **説明責任**が **持ちやすい**）。
- **`required_fields` 前提**: **必須列挙**が **「どの論理列・軸が揃っているか」**に **寄せやすく**、**粒度・期間・スコープ**は **別の仕組み・別リスト**で **足す**方が、**未固定の論点**（grain 正本など）と **衝突しにくい**（§6.4）。

**留保**: **time を論理フィールドとして必須にしたい**要件が **先に固まる**場合は **中間案**の **条件付き**を **再検討**しうる。**本項は「デフォルトの仮置き」**に過ぎない。

#### 3.5.3 path の外に置く候補

**「path に含めない方が自然ではないか」**の **候補**（**確定ではない**）。§3.4.3 と **重なるが、意味範囲の観点から再掲**する。

| 対象 | path の外に置くと整理しやすい理由（要約） |
|------|------------------------------------------|
| **`dataset_id`** | **データ境界**は **列・軸の識別**と **種類が異なる**。**path は dataset 内で解釈**と **規則化**し、**ID は別引数**にする方が **解釈がぶれにくい**場合がある。 |
| **`metadata_id`** | **分析メタ実体**への **参照**。**論理フィールド path のセグメント**とは **別カテゴリ**として **分離**しやすい。 |
| **`grain`** | **集計の解像度**は **「どの列があるか」とは別次元**。**path に押し込む**と **§5** の **正本位置未固定**と **相乗りで複雑化**しやすい。 |
| **`time_axis` の一部表現** | **期間型・as-of・階層**など **単一路に還元できない**表現（§3.3.3）。**path 文字列 1 本**に **無理に載せる**より、**構造化フィールド・複数セグメント・別軸**として **外に出す**方が **誠実**な場合がある。 |

**メタファの抑制**: **path を「全部入りの住所」**にしない方が、**004 / 006 / 013** の **責務**を **説明しやすい**（**過度に煽らず**、**範囲を広げすぎるコスト**を **認識する**程度のニュアンス）。

### 3.6 canonical key 仮置きメモ

**位置づけ**: **§3.3.2** の **id / key / name** 比較を **一段進め**、**dimensions / measures** を **将来の path や `required_fields` に載せる**ときの **識別子の仮軸**（**canonical key 候補**）を **整理**する。**正式命名規約・path syntax・最終優先順位の決定は行わない**。**JSON スキーマ・OpenAPI・backend 実装は行わない**。

**対象**: **`dimensions[]` / `measures[]` の要素識別**を **主**とする。**`time_axis`** は **単一路キーに落ちない**場合がある（§3.3.3）ため、**本節では軽く触れるに留める**。**time / grain / scope の正式識別子設計**は **全面展開しない**。

**本節で確定しないこと（再掲）**: **正式命名規約**。**path 文字列の syntax**。**id / key / name の最終優先順位**。**time / grain / scope** への **全面展開**。

#### 3.6.1 dimensions / measures の識別子候補

**004** の **`dimensions[]` / `measures[]`** 要素は、**少なくとも**次の **識別子系**を **含みうる**（**実装・データで欠ける場合あり**）。

| 対象 | 識別子候補 | メモ |
|------|------------|------|
| **`dimensions[]` 要素** | **`id`** / **`key`** / **`name`**（**他フィールド**も **ありうる**が **本メモの軸**は **この三者**） | **論理次元 1 件**を **安定して指す**ための **材料**。**canonical** は **別途** **仮置き**（§3.6.3） |
| **`measures[]` 要素** | **同上** | **論理指標 1 件**。**物理列・slot** との **対応**は **別表**（§3.3.1） |

**`time_axis`（参考）**: **単一キー**に **還元できる**場合は **path / 必須条件**の **軸**になりうるが、**dimensions/measures と同じノリで canonical を置けない**場合がある（§3.3.3）。**本節の主戦場外**。

#### 3.6.2 id / key / name の役割比較

| 観点 | **`name`** | **`key`** | **`id`** |
|------|------------|------------|----------|
| **内部参照に向くか** | **弱い**（**一意でない**場合がある） | **揃っていれば** **強い**（**スラッグ**） | **強い**（**永続・発番**の **意図**が **読みやすい**） |
| **表示に向くか** | **最も向く** | **向く**（**短い**） | **向きにくい**（**内部向け**） |
| **path / `required_fields` の足場に向くか** | **弱い**（**ラベル揺れ**） | **強い**（**セグメント名**に **寄せやすい**。**整備前提**が **要る**） | **強い**（**`dimensions.<id>` 等**。**スコープ**の **定義**が **要る**） |
| **人が読んで安定して解釈しやすいか** | **文脈依存** | **運用で一貫**すれば **安定** | **システム発番**なら **機械には安定** |
| **将来変更に強いか** | **弱い**（**リネーム**） | **中程度**（**スラッグ変更**は **影響**） | **強い**（**ID 不変**の **設計**が **前提**） |

**整理の芯**: **表示ラベル**（`name`）と **論理識別**（**canonical 候補**）と **内部実体参照**（`id`）を **混線させない**。

#### 3.6.3 現時点で安全な仮置き

**正式決定ではない**。**比較的安全**な **読み方の仮置き**（**後から変えうる**）。

| 役割 | 仮置き |
|------|--------|
| **`name`** | **表示向け**。**path / 必須列挙の主キーにしない**方が **誤読が少ない**（**表記ゆれ**）。 |
| **`id`** | **内部実体参照向け**（**永続 ID** の **意図**）。**外部公開**で **そのまま出すか**は **別途**（§3.6.5）。**path セグメント**に **使う**案も **自然**だが、**スコープ**（dataset 内等）を **先に** **揃えたい**。 |
| **`key`** | path / requirement 側の候補になりやすい（短い論理スラッグ）。**ただし**現状 **`key` が常に整備されている**とは限らない。整備が薄い場合、canonical の仮軸は **`id`** に寄せる検討が必要（運用負担とトレードオフ）。 |

**短文**: **`key` が揃っている**なら **`key`** を **論理 path / required の仮軸**に **しやすい**。**揃っていない**なら **`id`** を **仮軸**にし、**`key` 整備**を **並行**する、**という順序**が **現実的**になりうる（**断定しない**）。

#### 3.6.4 canonical key が必要になる場面

| 接続先 | なぜ必要か（要約） |
|--------|---------------------|
| **004 ↔ 006 対応表** | **観測要素**と **論理参照**を **1 列で** **対応**させる **軸**。**曖昧なラベル**だけだと **表が書けない**。 |
| **006 path 語彙** | **`dimensions.<…>` / `measures.<…>`** の **`<…>`** に **何を入れるか**の **たたき台**。 |
| **`required_fields`** | **必須列挙**が **同義異表記**で **分裂**しないよう、**canonical** を **共有**する。 |
| **OpenAPI 例示** | **フィールド例**が **実装者**に **ブレなく** **伝わる**か。 |
| **UI 表示・ログ** | **ユーザー向け文字**と **デバッグ用 ID** を **分ける**と **追跡**しやすい。**同一文字列**に **無理に寄せない**選択肢（§3.6.2）。 |

#### 3.6.5 実装前に確認したいこと

**canonical key 仮置き**を **実装・OpenAPI に落とす前**に **揃えたい**確認（**列挙**）。

- [ ] **`dimensions` / `measures` で `key` が常にあるのか**（**欠ける**場合の **頻度**）。
- [ ] **`key` がない場合**、**仮 canonical** を **`id`** と **`name`** の **どちらに寄せるか**（**§3.6.3**）。**両方混在**の **許容**。
- [ ] **`name` の表示揺れ**（**別名・多言語**）を **どこまで許容**し、**論理同一性**を **`id`/`key` で担保**するか。
- [ ] **`id` を外部公開**するか **内部限定**か（**ログ・URL** に **載せる**か）。
- [ ] **path と UI 表示**で **別識別子**を **許すか**（**表示は `name`**、**API は canonical** 等）。**混線**しない **説明**を **一文**で **残す**。

**次段（レビュー観点）**: **§3.6.3 の仮置き**を **正式採用する**のではなく、**何を見れば妥当性を判断しやすいか** — **§3.7**。

### 3.7 canonical key 仮置きのレビュー観点メモ

**位置づけ**: **§3.6.5** の **チェック項目**を **起点**に、**実データ確認**や **API ドラフト閲覧**（**OpenAPI ファイルの変更は行わない**）に **持ち込む前**の **レビュー観点**を **短く**整理する。**canonical key の正式決定・path syntax・OpenAPI 改稿は行わない**。

**何を確認するとよいか（観点の束）**:

| 観点 | 見る理由（要約） |
|------|------------------|
| **`key` の安定存在** | **dimensions / measures** で **どの程度** **`key` が揃いそうか**。**仮置き**（§3.6.3）の **前提**が **現実と合うか** |
| **`key` 欠落時の `id` 仮軸** | **`id` だけ**で **path / 必須列挙**を **組んでも破綻しないか**（**重複・スコープ**） |
| **`name` 揺れの漏れ** | **表示揺れ**が **path / `required_fields`** に **そのまま** **入っていないか** |
| **`id` の外部露出** | **公開**して **問題ない性質**か（**推測可能性**・**個人情報**等は **別論**） |
| **path 用と UI 用** | **論理識別子**と **表示ラベル**を **分ける必要**が **すでに** **見えているか** |

#### 3.7.1 実データ確認で見たいこと

**004 等のメタ JSON・実データ**を **見る**場合の **有効な確認**（**例**。**網羅ではない**）。

- **同一論理**なのに **`name` 表記が揺れていないか**（**別表記・略称・多言語**）。**揺れる**なら **`name` を canonical にしない**仮置き（§3.6.3）が **妥当**か **検証**できる。

## 5. grain / dimension / measure の正本候補位置

### 5.2 grain 論点との接続（対応表ドラフトから）

**結論を断定しない**。§3.3.1 の表を置くと、少なくとも次が **顕在化する**。

### 5.4 grain / time_axis 論点との接続（意味範囲から）

**§3.5 との接続**: **path の意味範囲**を **広げる**ほど、**grain** と **time_axis** を **同じ「path 記法」に押し込みたくなる**圧力が **生じうる**。

| **time と grain の置き場の仮説** | **time_axis を path のどこにマップするか**の **仮説が無い**と、**必須期間・必須粒度**が **004 と 006 のどちらを正とするか**で衝突しうる（§5.2）。**断定は不要**だが **論点の棚卸し**は有用 |

**一文メモ**: **`required_fields` は「足場」が揃ってから**が、**既存契約・§19.6 を壊しにくい**。

### 6.3 `required_fields` に必要な最小 path 前提（§3.4 との対応）

**スキーマ定義・OpenAPI 記述は本項に含めない**。**将来 `required_fields` を議論する**ときに、**§3.4 のドラフトが何を「最低前提」にしうるか**を整理する。

| 観点 | 内容（整理・候補） |
|------|---------------------|
| **最低限必要な語彙（骨格）** | **論理列・論理軸**を列挙するなら、**少なくとも dimensions / measures 系のセグメント**（例: `dimensions.<key>` / `measures.<key>` **のような形の候補**）が **説明の中心**になりやすい。**time・grain は「必須フィールドの 1 行に載せる」前に** **別合意**が要る場合がある（§5.3）。 |
| **一意性の前提** | **同一スコープ内**で **canonical key が一意**に解釈できること。**004 の id / key / name** の **どれを正とするか未定**だと、**必須列挙が同義異表記**で分裂しうる（§3.3.2・§6.2）。 |
| **canonical key の必要性** | **`required_fields` は論理参照**であり、**表示名だけ**では **機械的整合が取りにくい**。**「どのキーを path の根にするか」の初期方針**がないと **導入時に揉めやすい**（**正式規約の確定は不要**、**合意のたたき台**でよい）。 |
| **grain / time の未確定が効く箇所** | **必須性**を **「どの列」**に課すのか、**「どの粒度・どの期間」**に課すのかが **混線**しうる。**time と grain の置き場・二重定義**が **未固定のまま**だと、**必須セットの解釈**が **004 観測と 006 論理**で **食い違う**リスクがある（§5.2・§5.3）。 |
| **dataset / metadata との関係** | **`required_fields` の列挙**を **dataset で閉じるか**、**metadata スコープを別引数にするか**は **path 文字列だけでは決まらない**（§3.4.3）。**API 設計は別イテレーション**。 |

**再掲（境界）**: **`required_fields` は 006 側の論理参照の想定領域**。**013 evidence は observation trace のまま**（§19.6）。**本節は path 正本の確定ではない**。

### 6.4 `required_fields` に与える影響（意味範囲から）

**スキーマ定義は行わない**。**path の意味範囲**（§3.5）が **広いか狭いか**で、**将来の `required_fields` 議論**に **どう効くか**を **比較**する。

| path の意味範囲 | `required_fields` に **与えうる影響**（傾向） |
|-----------------|---------------------------------------------|
| **狭い**（**列・軸中心**） | **必須の主語**が **「どの論理列・論理軸が揃っているか」**に **寄りやすい**。**検証**は **キー一覧の包含**に **近づきやすい**。**粒度・期間・dataset 境界**は **path と別**に **「必須条件」**を **定義する必要**が **残る**（**欠けを別枠で補う**） |
| **広い**（**粒度・期間・スコープ込み**） | **1 つのリスト**に **列必須・粒度必須・期間必須・スコープ**が **混在**しうる。**優先順位**（**どれがブロック条件か**）の **説明**が **重くなる**。**004 観測**と **論理必須**の **差**が **同一文字列上に載る**と、**§19.6** の **trace と論理参照**の **説明**が **難しくなる**場合がある |

**短文の整理**: **狭い path** は **`required_fields` を「列・軸必須」に寄せやすい**。**広い path** は **表現は一括に見える**が、**必須条件の種類が混ざり**、**未固定の grain / time** の **論点を path に隠す**リスクがある。**議論に入る前に**（§3.5.2）、**意味範囲の仮置き**を **揃えた方が**、**後からの修正コスト**が **抑えられやすい**（**確定ではない**）。

### 6.5 `required_fields` 導入前提レビュー

**位置づけ**: **`required_fields` は未導入**であり、**本節は JSON スキーマ・OpenAPI 記述に入る前の「前提レビュー」**である。**急いで仕様化することを目的としない**。**§3.5 の狭義 path 仮置き**を受け、**何を `required_fields` に載せてよく、何を別表現へ逃がすべきか**の **整理**を **目的**とする。**万能箱**として扱わない。

#### 6.5.1 `required_fields` に載せやすいもの / 載せにくいもの

| 区分 | 対象 | 載せやすさと理由（要約） |
|------|------|---------------------------|
| **載せやすい候補** | **dimensions の個別論理対象** | **狭義 path**（§3.5）と **整合**しやすい。**「この論理次元が揃っていること」**を **キー列挙**に **近い形**で **表しやすい**（**canonical key の仮置き**が前提、§6.2）。 |
| **同上** | **measures の個別論理対象** | **論理指標**を **必須集合**として **列挙**しやすい。**検証**も **包含関係**で **説明しやすい**場面がある。 |
| **そのまま載せにくい候補** | **grain** | **列・軸とは種類が異なる**（§3.5.3・§5）。**正本位置未固定**のまま **必須リストに混ぜる**と、**「列が揃ったか」と「粒度が揃ったか」**が **同一語彙で衝突**しうる。 |
| **同上** | **time の一部条件** | **単一路 path に還元できない**表現がありうる（§3.3.3）。**期間・as-of・階層**が **混在**すると、**path 1 本**では **足りない**。 |
| **同上** | **dataset scope** | **データ境界**は **論理列のキー**と **軸が異なる**（§3.4.3）。**必須スコープ**は **別パラメータ・別フィールド**の方が **誤読が少ない**場合がある。 |
| **同上** | **`metadata_id`** | **分析メタ実体の参照**。**論理フィールド必須**とは **別カテゴリ**。**path 文字列と同一リスト**に **載せる**と **解釈が重層化**しやすい。 |

#### 6.5.2 `required_fields` が表したい条件の分解

**将来**、`required_fields` が **表しうる条件**を **種類**に **分けて**考える（**確定仕様ではない**）。

| 観点（分解） | 内容イメージ |
|--------------|--------------|
| **必須の列・軸** | **論理次元**が **揃っていること**（**dimensions 系**）。 |
| **必須の指標** | **論理指標**が **揃っていること**（**measures 系**）。 |
| **必須の時間条件** | **解釈可能な期間・基準日・階層**など（**単一 path に収まらない**可能性、§3.5.3）。 |
| **必須の粒度条件** | **集計の解像度**（**grain**。**列必須とは別次元**）。 |
| **必須のスコープ条件** | **どの dataset / メタ文脈**で **必須性を評価するか**（**path セグメント以外**）。 |

**「全部を `required_fields` 1 本で表すのが自然か」**の **比較**:

- **一括で載せる**案は、**API 上は短く見える**一方、**§3.5** で述べた **意味範囲の混在**と **同型のリスク**がある。**必須条件の種類が違うもの**を **同一リスト**に **無秩序に並べる**と、**優先順位・検証手順**が **説明しにくい**。
- **分解する**案は、**フィールド数は増える**が、**004 観測（trace）**と **006 論理要求**の **差**を **型として分けやすい**（§6.5.4）。

**本項の結論（断定しない）**: **自然さ**だけで選ぶと **一括**に **寄りやすい**が、**責務と検証**の観点では **分割・ハイブリッド**の **比較が必要**（次項）。

#### 6.5.3 path ベースで足りる範囲 / 足りない範囲

**前提**: **006 path は狭義寄り**（§3.5.2 仮置き）。**スキーマ確定ではない**。

| 範囲 | 内容 |
|------|------|
| **path ベースで自然に表現しやすい** | **dimensions / measures の個別対象**を **キー列挙**として **必須化**する意図。**「この論理参照が揃っている」**の **核**になりうる。 |
| **path だけでは不足しやすい** | **grain**: **集計粒度**は **列キーの包含**と **同じ検証**に **しにくい**。**time**: **複数の時間セマンティクス**が **単一 path 列**に **収まらない**可能性。**scope**（**dataset・metadata**）: **「どの文脈で必須か」**は **path の外**で **指定**した方が **混線が少ない**場合がある（§3.5.3）。 |

**短文**: **狭義 path** は **列・軸必須**の **表現に寄せやすい**。**粒度・時間・スコープ**は **別枠**を **検討**する余地がある（§6.5.4）。

#### 6.5.4 単一リスト案 / 分割案 / ハイブリッド案の比較

**いずれも正式決定ではない**。**比較表**として並べる。

| 案 | 内容 | 分かりやすさ | 誤読しにくさ | 004 / 006 / 013 / 将来 OpenAPI との相性（傾向） |
|----|------|--------------|--------------|--------------------------------------------------|
| **A. 単一リスト案** | **`required_fields` に**列・軸・粒度・時間・スコープ**を **載せようとする** | **クライアントは 1 フィールドを見ればよい**ように **見える** | **中身の種類が混ざり**、**「必須」の意味が行ごとに違う**と **誤読**しやすい | **004** の観測 JSON を **そのまま並べた**ように **見えるリスク**。**013** の **trace** と **論理必須**の **差**が **説明しづらい**。**OpenAPI** でも **1 配列に多型**が **詰まり**やすい |
| **B. 分割案** | **`required_fields` は列・軸中心**に **絞り**、**粒度・時間・スコープ**は **別表現**（**別キー・別オブジェクト**等。**未定義**） | **フィールドが増え**、**読者の負担**は **増えうる** | **必須の種類**が **分離**され、**検証ルール**を **分けて書きやすい** | **004** 観測と **006 論理**の **レイヤ**を **揃えやすい**。**013** は **evidence は trace**のまま（§6.5.5）。**OpenAPI** は **型を分けやすい**（**具体形は未着手**） |

**比較メモ**: **万能箱を避ける**なら **B または条件付き C** の **議論**が **自然**。**A は短く見える負債**になりうる、という **警戒**に留める（**断定しない**）。

#### 6.5.5 evidence と logical requirement の違い

**将来 `required_fields` を導入しても**、次の **区別**を **維持**することが **前提レビューの結論側**になりうる。

| 項目 | **013 `evidence`（observation trace）** | **`required_fields`（論理要求の表現候補・未導入）** |
|------|----------------------------------------|---------------------------------------------------|
| **役割** | **004 観測の説明**。**候補生成の文脈**（§19.6） | **整合・I/O** で **「揃えるべき論理参照」**を **示す候補**（**契約・検証に近い**語彙になりうる） |
| **正本** | **path 正本ではない** | **006 側の論理表現**として **議論される**。**evidence の写しを正本にしない** |
| **混同のリスク** | **同じ文字列**が **trace と必須列挙**の **両方に出る**と、**「観測された」**と **「要求される」**が **同一視**されうる | **仕様化時**に **説明**で **明示**する必要 |

**一文**: `required_fields` は **logical / contractual な要求表現の候補**として **限定的に捉え**、**013 evidence は trace** のまま、**役割を分ける**。

#### 6.5.6 導入前チェック項目

**`required_fields` を仕様化（スキーマ化）する前**に、**少なくとも**次を **確認した方がよい**前提として **列挙**する。**チェック済み＝実装済みではない**。

- [ ] **path の意味範囲**が **文書上すり合っているか**（§3.5。**狭義寄り仮置き**の **共有**）。
- [ ] **canonical key** の **仮置き**（id / key / name の **どれを path の根にするか**の **たたき台**）があるか（§3.3.2・§6.2）。
- [ ] **grain** を **path に含めるか・別表現か**が、**未整理のまま** **必須議論に突入**していないか（§5・§6.5.1）。
- [ ] **time 条件**の **表現単位**（**期間・as-of・階層**の **どれを必須にするか**）が **整理されているか**（**単一 path に落ちるか**含む）。
- [ ] **scope**（**dataset・metadata**）と **path** の **混線**がないか（**必須性の評価文脈**が **明示**できるか）。
- [ ] **evidence**（**観測 trace**）と **logical requirement**（**論理必須**）の **区別**が **仕様の説明に明記**される前提になっているか（§6.5.5）。
- [ ] （**任意**）**005 / 011** の **正本**と **`required_fields`** の **関係**を **先走りでゲートにしない**合意（**gating 未導入**の趣旨と **整合**）。

### 6.6 `required_fields` 分割案の別フィールド責務ドラフト

**位置づけ**: **§6.5** の **分割案・ハイブリッド案**を **前提**に、**`required_fields` 以外に置く候補**について **「何を表し、何を表さないか」**を **責務レベル**で **整理**する。**JSON スキーマ・正式フィールド名の確定は行わない**。**仮ラベル**で **比較**する。**採用決定ではない**。

**目的**: **`required_fields` を軽くするために何を外へ出すか**を **明示**し、**外へ出した先の責務**を **曖昧にしない**（**単一リスト案の合理性**も **§6.5.4** で **残している**。**分割案を押しつけない**）。

#### 6.6.1 分割対象の候補

以下は **候補の束**である。**名称は仮**（**`required_*` は比喩**。**確定しない**）。

| 仮ラベル（候補） | 主に担う内容（イメージ） | §6.5 との関係 |
|------------------|--------------------------|----------------|
| **`required_fields` 相当** | **列・軸・指標**（**dimensions / measures 系**）を **論理参照として必須化**する **中心リスト** | **狭義 path**（§3.5）と **整合**しやすい **核** |
| **`required_grain` 的なもの** | **集計の解像度・単位**など、**「どの粒度で揃っていること」**を **必須化**する **別表現** | **grain** を **`required_fields` に混ぜない**余地（§6.5.1） |
| **`required_time` 的なもの** | **期間・as-of・階層**など、**時間セマンティクス**を **必須化**する **別表現** | **time** を **単一路 path** に **還元できない**場合の **受け皿**（§3.3.3） |
| **`required_scope` 的なもの** | **dataset 境界・metadata 文脈**など、**「どのスコープで必須を評価するか」** | **path セグメント以外**の **必須条件**（§3.5.3・§6.5.1） |

**補足**: **実際の API** では **1 オブジェクトに複数キー**を **まとめる**案も **比較対象**になりうる。**本項は「論理ブロック」の切り方**を **先に**揃える。

#### 6.6.2 各候補の責務と非責務

**読み方**: 各行は **「そのブロックがあれば何が説明できるか」**の **たたき台**。**実装済みではない**。

| 仮ラベル | 何を **表す**ためのものか | 何を **表さない方がよい**か | **path**（狭義）との関係 | **013 evidence** との違い | **005 / 011** との関係 |
|----------|---------------------------|----------------------------|--------------------------|---------------------------|-------------------------|
| **`required_fields` 相当** | **論理列・論理軸・論理指標**が **揃っていること**（**キー列挙・整合**） | **粒度・期間・データ境界そのもの**を **無理に同一リストに詰める**こと（§6.5.1）。**「観測された」事実の主張** | **006 の logical path 文字列**（**狭義**）と **対応**。**検証**は **キー包含**に **寄せやすい** | **trace** は **004 観測の説明**。**こちらは requirement**（§6.5.5） | **review / suppression**（**005**）や **補助評価**（**011**）の **正本ではない**。**必須性の論理**と **混同しない** |
| **`required_grain` 的** | **分析・集計が解釈可能な粒度**が **満たされていること**（**必須粒度条件**） | **個別列の有無**まで **同一フィールドで肩代わり**すること（**列必須は `required_fields` 側**）。**grain の正本**を **ここで確定する**こと（**未決**、§5） | **path の外**。**「列キー」とは別次元**の **必須** | **観測された time_axis / grain** の **写しではない**。**要求** | **011 のリスク**や **005 の unresolved** を **grain 必須の充足**と **同一視しない**（**評価・抑制は別レイヤ**） |
| **`required_time` 的** | **期間・基準日・階層**など **時間条件**が **満たされていること** | **全てを 1 文字列 path** に **押し込む**こと。**複数の時間概念**を **無区別に並べる**こと | **単一の `time.<key>`** に **落ちる場合のみ** path と **近い**。**落ちない**なら **構造化**が **主** | **004 `time_axis` の観測 JSON** ではない | **同上**。**時間の妥当性**を **011 スコア**と **混線させない** |
| **`required_scope` 的** | **どの dataset / どの metadata 文脈**で **上記の必須を評価するか** | **論理列そのものの ID** を **scope だけで代替**すること。**path の置換** | **path** は **通常そのスコープ内で解釈**。**scope は path の親・文脈** | **trace に載る dataset_id / metadata_id** の **写しではない**（**要求側の境界**） | **005 が抑止した列**と **「必須だが欠けている」**を **同じ語で扱わない**合意が **要る**（**別イテレーション**可） |

#### 6.6.3 grain / time / scope を `required_fields` に入れない（別表現にする）利点

| 観点 | 内容（整理） |
|------|--------------|
| **解釈の混線を避けやすい** | **「列が揃った」**と **「粒度が揃った」**と **「期間が揃った」**を **別ブロック**に **分ける**と、**必須判定**の **主語**が **説明しやすい**（§6.5.2）。 |
| **path の意味範囲を保ちやすい** | **狭義 path**（§3.5）を **`required_fields` 相当**に **寄せられる**。**path 文字列**が **万能キー**に **膨らみにくい**。 |
| **trace と requirement の違いを保ちやすい** | **観測**（013 evidence）と **論理必須**（各 `required_*` 相当）の **差**を **フィールド境界**で **示しやすい**（§6.5.5）。 |
| **将来の OpenAPI / UI で説明しやすい** | **ラベル・ヘルプ**を **ブロック単位**で **書き分けられる**。**「このブロックは論理必須、あちらは観測」**が **画面設計**に **載せやすい**（**具体形は未着手**）。 |

#### 6.6.4 分割しすぎる場合の注意点

**分割案の長所**（§6.5.4）と **併せて**、**負担**も **整理**する。**分割を推すだけではない**。

| 注意点 | 内容 |
|--------|------|
| **フィールド数が増える** | **クライアント・ドキュメント**の **見通し**が **一瞬悪化**しうる。**「まずは 2 ブロック（`required_fields` + scope）だけ」**など **段階導入**の **余地**は **比較対象**。 |
| **相互関係の説明が必要** | **「列必須だけ満たし、粒度必須が欠ける」**ときの **意味**。**合成規則**（§6.6.6）が **ないと** **解釈が分岐**する。 |
| **empty / omitted / unknown** | **ブロックごと**に **「未指定＝不要」**なのか **「未指定＝検証不能」**なのかが **揃っていない**と、**実装と説明**が **食い違う**。**単一リスト案**は **その点では短く見える**（**代償**は §6.5.4）。 |

#### 6.6.5 現時点での最小責務仮置き

**以下は採用決定ではなく、議論の仮置き**である。

- **`required_fields` 相当**: **列・軸・指標**（**dimensions / measures 系の論理参照**）を **中心**に **必須化**する **責務**に **寄せる**。
- **`required_grain` 的・`required_time` 的・`required_scope` 的**: **`required_fields` に無理に詰めない**。**別表現候補**として **並べて**、**それぞれの非責務**（§6.6.2）を **外に出す**。

**短文**: **最小の安全そうな切り方**は、**論理列・軸は path 系の必須リスト**、**粒度・時間・スコープは別ブロック**、という **分離**。**ハイブリッド**（§6.5.4 **C**）は **time だけ隣接**等 **条件付き**で **再検討**しうる。**確定ではない**。

#### 6.6.6 なお必要な前提

**本節の責務整理をしたうえで**、**なお** **仕様化前**に **揃えたい**前提（**列挙**。**未充足のまま** **JSON 化**すると **負債**になりうる）。

| 前提 | 内容（整理） |
|------|--------------|
| **canonical key 仮置き** | **path の根**に **どのキー**を **使うか**（§3.3.2）。**無いと** **ブロック間**で **同じ列を別名**で **二重必須**に **見える**。 |
| **path 対象と scope 対象の切り分け** | **論理列**は **`required_fields` 相当**、**データ境界・メタ文脈**は **`required_scope` 的**。**混在**させる **ルール**を **先に** **文書化**。 |
| **grain の正本位置** | **どのブロックが grain を **正**と **みなすか**（§5）。**未固定のまま** **2 ブロックに grain が潜む**と **二重定義**。 |
| **time 条件の粒度** | **期間・as-of・階層**の **どれを必須にするか**。**単一キー**に **落ちるか**（§3.3.3）。**`required_time` 的**の **内部構造**は **別合意**。 |
| **別フィールド相互の合成規則** | **AND** なのか **優先順位**があるのか、**片方が空**のときの **意味**。**§6.6.4** の **empty / omitted** と **直結**。**意味面のドラフト**は **§6.7**。 |

### 6.7 required 条件ブロックの合成規則ドラフト

**位置づけ**: **§6.6** の **仮ラベル**（`required_fields` 相当 / `required_grain` 的 / `required_time` 的 / `required_scope` 的）を **複数持つ**場合、**全体として requirement をどう満たすか**の **読み**を **先に**揃える。**JSON 形・スキーマ定義は行わない**。**実装済みではない**。

**目的**: **分割した requirement** を **どう組み合わせて解釈するか**（**意味の合成**）。**単純化しすぎず**、**複雑化しすぎない**。**trace / evaluation / suppression** と **混同しない**。

#### 6.7.1 最小合成規則（AND 中心）

**最小仮置き（正式採用ではない）**: **複数の required 条件ブロック**が **同時に**与えられるとき、**まずは最も保守的な読み**として **全体は AND** とする。

- **例**: **`required_fields` 相当**があり、**`required_time` 的**があり、**`required_scope` 的**があるなら、**それぞれのブロックが示す条件をすべて満たして初めて**、**論理 requirement 全体を満たす**、と **読む**（**各ブロックの内部**が **どう検証されるか**は **別論**、§6.6）。
- **意図**: **片方だけ満たせばよい**と **誤読**されるのを **避けやすくする**。**OR や優先順位**は **§6.7.3** で **軽く**触れる。

**留保**: **AND 固定**が **常に正しい業務意味**とは **限らない**。**将来** **別の合成**が **要る**場合は **§6.7.3** の **拡張余地**を **前提**にする。

#### 6.7.2 合成で問題になりやすい点

| 論点 | 内容（整理） |
|------|--------------|
| **ブロックが「空」のとき** | **空配列・空オブジェクト**が **「このカテゴリの必須は無い」**なのか、**「必須集合が空＝何も要求しない」**なのか、**「検証不能」**なのかで **解釈が分岐**しうる。**ブロックごと**に **定義が要る**（§6.7.4）。 |
| **ブロックが omitted のとき** | **キー自体が無い**場合、**「そのカテゴリの requirement は適用しない」**なのか **「デフォルトで AND の片側が欠けている」**のか。**AND 仮置き**と **組み合わせる**と **特に**説明が **要る**。 |
| **unknown / undecided 的な状態** | **システムがまだ評価できない**ときに **必須判定**を **どう表示するか**。**requirement オブジェクトに載せるか**、**別ステータス**にするかは **未固定**。**無理に** **empty と同一視**しない方が **安全**な場合がある。 |
| **片方だけあるとき** | **例**: **`required_fields` のみ**。**他ブロック omitted**。**AND 仮置き**では **「他は無条件」**と **読む**か **「他は未指定で不充足」**と **読む**かで **実装が分岐**する。**仕様化前**に **一文**で **決めたい**。 |
| **grain / time / scope が `required_fields` の代替に見える誤読** | **列必須**を **`required_grain` 的**だけで **満たした気分**になる、**`required_scope` 的**が **path の省略**と **混ざる**、など。**§6.6.2 の非責務**を **合成の説明**でも **繰り返す**と **抑えやすい**。 |

#### 6.7.3 AND / OR / 優先順位の比較メモ

**今回の主軸は AND**。**以下は比較・将来拡張余地**。**詳細仕様化は行わない**（**やらないこと**に従う）。

| 読み | 内容（要約） | 今回の位置づけ |
|------|--------------|----------------|
| **AND のみ** | **全ブロック**の **条件を満たす** | **§6.7.1 の最小仮置き**。**説明が単純**。**誤読**（**片方で足りる**）を **抑えやすい** |
| **一部 OR** | **同一カテゴリ内**や **ブロック間**で **選択肢**を **許す** | **業務**で **「期間 A または B」**などが **要る**場合の **拡張余地**。**複雑性**と **トレードオフ**。**今回は触れるのみ** |
| **優先順位つき評価** | **先に scope**、**次に time** など **順序**で **判定** | **ゲート的**に **見える**ため **§8（gating）** や **policy** と **混線**しやすい。**gating 未導入**の **現状**では **慎重**。**将来**の **別合意**として **置ける** |

#### 6.7.4 empty / omitted / unknown の意味

**断定しすぎない**。**API の型**は **未確定**。**「どれに近いか」**を **比較**する。

| 表現 | **requirement なし**に近いか | **未指定**に近いか | **未整理・未評価**に近いか | メモ |
|------|------------------------------|--------------------|----------------------------|------|
| **empty list / empty object** | **場合による** | **場合による** | **稀に** | **「必須集合が空」**は **論理的には** **要求なし**に **近い**が、**「空はエラー」**と **読む**設計も **ありうる**。**ブロックごと**に **決める** |
| **omitted（キー無し）** | **場合による** | **近い**ことが多い | **場合による** | **「そのブロックはスキーマ上オプション」**なら **未指定**。**AND 仮置き**では **「他ブロックだけで判定」**と **読む**か **要る**（§6.7.2） |
| **unknown / not_evaluated 的** | **通常は近くない** | **近くない** | **近い** | **データが足りない**・**まだ解析していない**等。**requirement 充足**とは **別次元**の **状態**として **分離**しやすい |

**重要**: **empty** と **omitted** を **安易に同一視**しない。**unknown** を **「必須欠落」**と **混同**しない（**§6.7.5**）。

#### 6.7.5 evidence / suppression / recommendation との違い

**合成規則**は **論理 requirement ブロック同士**の **読み方**である。**次と混同しない**。

| 対象 | 役割の要約 |
|------|------------|
| **013 `evidence`（observation trace）** | **004 観測の説明**。**required ブロックの充足判定**ではない（§6.5.5）。 |
| **005 suppression / review 正本** | **レビュー・抑制**の **正本**。**「必須が満たされたか」**の **論理**と **同じ語で** **結論**を **置き換えない**。 |
| **011 recommendation 等** | **補助評価**。**005 を上書きしない**。**requirement 合成の結果**と **同一視**しない。 |

**短文**: **requirement**（**将来の required 系**）は **「揃えるべき論理条件」**。**trace / suppression / recommendation** は **別レイヤ**。**合成規則**を **定義しても** **この境界は変わらない**。

#### 6.7.6 現時点で安全な最小読み

**仮置きのまとめ**（**採用決定ではない**）。

1. **分割ブロック**は **まず AND 中心**で **読む**（§6.7.1）。**OR・優先順位**は **後段**（§6.7.3）。
2. **empty と omitted** は **慎重に分ける**。**unknown** を **別カテゴリ**として **扱う余地**を **残す**（§6.7.4）。
3. **requirement** と **013 trace / 005 suppression / 011 recommendation** を **混同しない**（§6.7.5）。
4. **grain / time / scope** を **`required_fields` の代替**と **読まない**（§6.7.2）。

### 6.8 required 条件ブロックの状態意味メモ

**位置づけ**: **§6.7** の **合成規則（AND 中心・最小仮置き）**を **前提**に、**各ブロック**が **empty / omitted / unknown** のとき **どの意味に近いか**を **実装前**に **揃える**。**JSON スキーマ・OpenAPI・正式命名は確定しない**。**unknown の正式データモデル決定は行わない**。

**目的**: **後で実装や OpenAPI に落とす前**に **状態解釈のぶれ**を **減らす**。**「条件なし」と「未整理」を分ける**。**unknown を empty に畳まない**（§6.7.4）。

#### 6.8.1 対象ブロックと状態

**対象ブロック**（**§6.6** の仮ラベル。**正式フィールド名は確定しない**）:

| 仮ラベル | 指すブロック（再掲） |
|----------|----------------------|
| **`required_fields` 相当** | **列・軸・指標**（**狭義 path** に **整合**しやすい **論理必須**） |
| **`required_grain` 的** | **必須の粒度条件** |
| **`required_time` 的** | **必須の時間条件** |
| **`required_scope` 的** | **必須のスコープ条件**（**dataset・metadata 文脈** 等） |

**状態**（**意味候補**として **扱う**）:

| 状態 | 読み方のメモ |
|------|--------------|
| **empty** | **空配列・空オブジェクト**等、**構造はあるが中身が空** |
| **omitted** | **キー自体が無い**・**フィールドが送られていない** |
| **unknown / undecided / not_evaluated 的** | **まだ判定できない**・**入力不足**・**評価前** 等（**表現は未固定**） |

#### 6.8.2 empty / omitted / unknown の意味候補

**各状態**が **次のどれに近いか**（**断定しない**。**複合**もありうる）。

| 状態 | requirement なし | 未指定 | 未整理 / 未評価 | 判定保留 |
|------|------------------|--------|-----------------|----------|
| **empty** | **近いことが多い**（**特に**「必須キー集合が空」） | **場合による**（**スキーマ上は必須だが空」**の **矛盾**も **起きうる**） | **近くない**ことが多い | **場合による** |
| **omitted** | **場合による** | **近いことが多い** | **近いことがある**（**「まだ決めていない」**と **読める**） | **場合による** |
| **unknown 的** | **通常は近くない** | **近くない** | **近い** | **近い** |

**補足**: **empty** は **「このカテゴリの必須は無い」**と **読みやすい**一方、**業務上「空は不正」**と **読む**場合も **ある**。**omitted** は **未指定**と **未整理**の **中間**に **見える**ことが **ある**（§6.8.3）。

#### 6.8.3 ブロックごとの差

**一律の意味にしない**。**同一の empty でも** **ブロックで読みが分岐**しうる。

| ブロック | **empty** | **omitted** | **unknown 的** |
|----------|-----------|-------------|----------------|
| **`required_fields` 相当** | **「必須とする論理列・軸が無い」**に **寄りやすい**（**requirement なし**に **近い**）。**ただし** **「空は不正」**と **する**ポリシーも **ありうる** | **「列・軸の必須集合が未指定」**に **寄りやすい**。**AND 合成**では **他ブロックと** **「未指定＝無条件」**か **要検討**（§6.8.4） | **canonical key が未確定**で **列挙できない**等。**「必須欠落」**と **混同しない**（§6.7.4） |
| **`required_grain` 的** | **「粒度の必須制約が無い」**に **読める**。**grain 正本未固定**の **まま** **空**と **読むと** **二重定義**と **衝突**しうる（§5） | **「粒度必須をまだ課していない」**と **「課すか未整理」**の **両方**に **見える**。**後者**は **未整理**に **近い** | **集計解像度が判定不能**。**業務上** **「まだ確定していない」**の **含み**を **持ちやすい** |
| **`required_time` 的** | **「時間条件の必須が無い」**。**期間・as-of・階層**の **どれも要求しない**に **近い** | **「時間必須を課さない」**と **「課すが中身未定」**の **差**が **つきにくい**。**未整理**に **読まれる**リスク | **単一路 path に落ちない**（§3.3.3）。**評価不能**に **近い** |
| **`required_scope` 的** | **「スコープ制約が無い」**＝ **グローバル／デフォルト文脈**と **読める**場合と **危険**（**意図しない全体適用**） | **「どの dataset / metadata で評価するか未指定」**。**ビジネス上** **致命的な曖昧さ**に **なりやすい** | **データ境界が未確定**。**unknown** は **「どの契約文脈で必須か決められない」**に **近い**ことがある |

**要点**: **`required_fields` の empty** は **「必須列なし」**に **寄りやすい**。**`required_time` の omitted** は **「時間条件なし」**と **「未整理」**が **紛れやすい**。**`required_scope` の unknown** は **業務上** **文脈未定**の **重さ**を **持ちやすい**。

#### 6.8.4 合成規則との接続（§6.7）

**AND 中心**の **最小仮置き**（§6.7.1）と **組み合わせた**ときの **読み**（**一例**。**確定ではない**）。

| 読み方 | **全体解釈がどう変わりうるか** |
|--------|--------------------------------|
| **empty を requirement なし** と **読む** | そのブロックは **AND の束**から実質的に外れる（「条件が無い」）。**他ブロックだけ**で充足を判定 |
| **omitted を未指定** と **読む** | **AND** の要素が欠けているのか無関係なのかが一文で決まっていないと、「不充足」と「未指定でスキップ」が分岐（§6.7.2） |
| **unknown を保留・未評価** と **読む** | **全体**の **requirement 充足**が **真偽ではなく「未確定」**になりうる。**AND** なら **1 ブロックでも unknown** で **全体が unknown** になりうる **読み**が **自然**な場合がある（**別案**: **unknown は欠落**と **する**。**仕様で** **揃える**） |

**短文**: **empty・omitted・unknown** の **解釈**が **ブロックで違う**と、**AND** の **全体**の **意味**が **変わる**。**§6.7** と **矛盾しないよう** **ブロック別**に **一文**ずつ **残す**と **安全**。

#### 6.8.5 現時点で安全な仮置き

**正式決定ではない**。**誤読が少なそうな読み**の **仮置き**。**全ブロック同一**とは **しない**。**unknown を empty に畳まない**。

| ブロック | **empty**（仮置き） | **omitted**（仮置き） | **unknown**（仮置き） |
|----------|---------------------|----------------------|------------------------|
| **`required_fields` 相当** | **「このカテゴリでは論理列・軸の必須は課さない」** | **「論理列・軸の必須集合が未指定」**（**AND との関係は §6.8.4**） | **「列キーが確定せず、必須集合を言えない」**。**empty にしない** |
| **`required_grain` 的** | **「粒度の必須制約は課さない」** | 「粒度必須を課すかまだ決めていない」に見えるなら未整理として別扱いを検討 | **「解像度が判定不能」**。**保留** |
| **`required_time` 的** | **「時間の必須制約は課さない」** | 「時間必須を課すか・中身は未整理」の切り分けを要る | **「時間セマンティクスが評価不能」**。**保留** |
| **`required_scope` 的** | **慎重**。**「スコープ無指定＝デフォルト」**が **危ない**なら **empty を許さない**ポリシーも **比較対象** | **「評価文脈が未指定」**。**業務上** **危険**なら **不許可**と **読む**か **要検討** | **「どの dataset / metadata で評価するか未定」**。**保留** |

#### 6.8.6 実装前に確認したいこと

**OpenAPI・実装に入る前**に **揃えたい**確認事項（**列挙**）。

- [ ] **各ブロック**で **empty / omitted / unknown** を **同じ意味にするか**、**分けるか**（§6.8.3）。
- [ ] **omitted** を **「送信省略」**（**クライアント都合**）と **論理未指定**（**まだ課さない**）の **どちらに主に寄せるか**。
- [ ] **unknown** を **requirement オブジェクト内**に **持つか**、**外**（**ステータス・ラップ**）に **持つか**。
- [ ] **UI** で **「条件なし」**と **「未整理」**が **見分けられるか**（**unknown** の **表示**）。
- [ ] **AND 合成**時に **1 ブロックが unknown** だと **全体**を **unknown** と **するか**、**欠落**と **するか**、**他ブロックで継続**するか。

### 6.9 required 条件ブロックの状態意味 合意メモ（実装前たたき台）

**位置づけ**: **§6.8** の **状態意味**を **確認しやすい最小形**に **圧縮**した **合意メモ**である。**新しい論点の追加**は **しない**。**最終決定ではない**。**実装前レビュー**で **参照するためのたたき台**。

#### 6.9.1 ブロック別 状態意味まとめ表

| ブロック（仮ラベル） | **empty** の現時点の読み（仮置き） | **omitted** の現時点の読み（仮置き） | **unknown** の現時点の読み（仮置き） | **誤読しやすい点** | **実装前に要確認の点** |
|----------------------|-----------------------------------|-------------------------------------|-------------------------------------|--------------------|-------------------------|
| **`required_fields` 相当** | **論理列・軸の必須を課さない**（必須集合なし） | **必須集合が未指定** | **列キー未確定のため必須集合を言えない**。**empty にしない** | **empty** を **「不正」**と **読む**ポリシーとの **衝突** | **AND** 時 **omitted** を **無条件**と **読むか**（§6.8.4） |
| **`required_grain` 的** | **粒度の必須制約を課さない** | **課すか未決**に **見える**と **未整理** | **解像度が判定不能**・**保留** | **grain** を **列必須の代替**と **見る** | **正本未固定**との **整合**（§5） |
| **`required_time` 的** | **時間の必須制約を課さない** | **「条件なし」**と **「中身未整理」**が **紛れうる** | **時間セマンティクスが評価不能** | **単一路 path** に **無理に落とす** | **期間 / as-of / 階層**の **どれを必須にするか** |
| **`required_scope` 的** | **スコープ制約なし**（**デフォルト文脈**に **読むと危ない**場合あり） | **評価文脈が未指定** | **dataset / metadata 境界が未定** | **スコープ省略**を **全体適用**と **誤る** | **empty を許すか**、**未指定をエラーにするか** |

#### 6.9.2 状態意味の短い原則

- **empty** は **常に omitted と同じ意味ではない**（**構造はあるが空** vs **キー無し**）。
- **omitted** は **常に「条件なし」ではない**（**未指定**・**未整理**に **近い**読みが **混じりうる**）。
- **unknown** は **empty に畳まない**（**未確定・保留**。**必須欠落**と **混同しない**）。
- **ブロック差**を **残したまま**扱う（**4 ブロックで同一ルール**に **しない**）。

#### 6.9.3 合成規則との最小接続（§6.7）

- **AND 中心**の **仮置き**（§6.7.1）では、**ブロックごとの状態意味の差**が **全体の充足・未確定**の **解釈に効く**。
- **特に unknown** は **全体を未確定**と **読む**など **AND** の **外れ値**になりうる（§6.8.4）。**別案**（**欠落**扱い）も **仕様で揃える**必要がある。
- **状態意味**を **揃えないまま** **API 化**すると **省略値**の **解釈が実装ごとにズレる**。**意味を先に** **合意**した方が **安全**。

#### 6.9.4 実装前レビューでの使い方

**本合意メモ**は **次の確認**に **使う**（**網羅的なチェックリストではない**）。

| 用途 | 内容 |
|------|------|
| **JSON 形状レビュー前** | **empty / omitted** を **同型で表すか** **決める前**に、**§6.9.1** で **意味が揃っているか** |
| **OpenAPI 化前** | **必須・任意・欠落**の **語**と **§6.9.2** の **原則**が **矛盾しないか**。**ドラフト閲覧**は **§7** |
| **UI 文言** | **「条件なし」「未指定」「未確定」**の **ラベル**が **§6.9.1** と **対応**するか |
| **backend 実装** | **デフォルト値・省略**の **解釈**が **§6.9.1・§6.9.3** と **一致**するか |

**再掲**: **013 evidence** は **trace**。**本メモ**は **logical requirement** の **状態**の **話**（§6.7.5）。

## 8. gating / 表示レベル設計を後段に送る理由

### 8.1 現状

- **材料**はある: 005 suppression / unresolved 観測、011 の `decision_recommendation` / `risk_signals` 等。
- **していないこと**: それらを **サーバ側の単一ラベル**（例: blocked / limited / visible）に **変換して返す**こと。

### 8.2 後段に送る理由（整理）

1. **責務が早い**: gate 結果は **policy 層**に見えやすく、**013 が read-only 参照**という現在の境界と衝突しやすい。
2. **013 の土台がまだ stub**: **候補そのもの**が本格化していない段階で **表示レベルだけ先に固める**と、**利用者が「確定済みパイプライン」と誤解**しやすい。
3. **004/006/path が先**: **「何を指すか」**が揃わないと、**gate の入力が観測なのか意味確定なのか**が曖昧になる。
4. **SPEC-TI-014 §20** が **未着手**として gating を明示している。**本メモはそれに従う**。

## 14. T2 TableScopeSummaryView 追加点の一文定義メモ

**位置づけ**: **§12.T2**（**追加点の意図**）を **`TableScopeSummaryView`**（**§13.4**）に **当てはめた** **一文定義**の **たたき台**である。**serializer・view の変更はしない**。**意味**で **足す／足さない**を **先に** **揃える**。

**現状の事実（実装）**: 応答は **`table_id`** と **`refs`**。**`refs`** は **`get_latest_artifact_refs_for_table`** により、**最新 `NormalizedDataset` 列**に基づく **参照束**（**`dataset_id` / `metadata_id` / `evaluation_ref` / `session_id` / `suggestion_run_ref`** 等、**欠く場合は空文字**）が **入る**。**データセット無し**時は **`table_id`** のみに **近い形**。**同一テーブル**に **別 GET** で取る **`table_read_artifact`・`judgment_result` の latest** は、この **束には含まれない**（**`services.collect_artifact_refs_bundle_for_dataset` のスコープ外**）。

### 14.1 今回足すものの性質

**Phase 1 で足してよいもの**は、次を **同時に** **満たす** **観測・説明の補強**に **限定**する。

| 条件 | 内容 |
|------|------|
| **read-only** | **GET** の **返却**のみ。**DB 正本**を **更新しない** |
| **観測または説明の補強** | **呼び出し側**が **次にどの read-only URL を叩くか**・**何が「ある／まだない」か**を **誤解しにくくする**情報（**ペイロードの意味確定**ではない） |
| **正本状態の再定義をしない** | **005 / 011 / 013** の **意味**を **このレスポンスが決めない**。**「ここに載っているから正」**と **読めない**粒度 |
| **required / gating / recommendation の代替をしない** | **logical requirement**・**ゲート**・**011 推奨**の **窓**として **機能しない**。**判断材料の集約**に **見えない** |
| **Table summary として自然** | **テーブル単位**の **インデックス**。**詳細リソースの中身**ではない |
| **既存レスポンスを大きくしすぎない** | **詳細 JSON の埋め込みなし**。**キー数・文字列長**の **増分は小さい** |

### 14.2 refs に載せる候補

**`refs` は既に「導線用の参照束」**である。**追加点**も **同じ性格**に **寄せる**のが **Phase 1 では安全**（**別トップレベルキー**に **分ける案**は **T3** で **比較可**。**今回は候補の比較**のみ）。

| 候補 | 内容 | Phase 1 向き | 注意 |
|------|------|--------------|------|
| **A. table 直結 latest の ID（空文字可）** | **`judgment_result` / `table_read_artifact`** の **latest 行**に対応する **ID 文字列**を **`refs` 互換**の **「欠ければ空」**で **足す** | **高** | **001/002 の器**への **導線**。**既存の `evaluation_ref` 等と同じ「存在参照」パターン**に **揃えやすい** |
| **B. 存在だけのフラグ** | **`has_read_artifact` / `has_judgment`** のような **真偽**のみ | **中** | **ID より軽い**が **次 GET の URL 組み立て**には **別途 `table_id` だけ**で **足りる**ため **重複感**。**014 互換の string 束**と **並べにくい** |
| **C. 既存束の説明コメントに近いメタ（非推奨寄り）** | **`refs_provenance`** のような **「最新データセット由来」**の **短文** | **低〜中** | **説明力**は **上がる**が **文言**が **仕様に見えやすい**。**Phase 1 は避けてもよい** |

**比較の結論（仮置き）**: **A** を **第一候補**とする。**理由**: **既存 `refs` のキーがすべて「参照用 ID・空なら未生成」**という **一貫した観測**であり、**read-artifact / decision** も **同じ GET ファミリー**（**`/tables/{table_id}/...`**）への **軽い導線**として **並べる**のが **Table summary の役割**に **合う**。**B** は **キー増**の **割に** **得る情報**が **限定的**なら **採用優先度は下げる**。

### 14.3 refs に載せないもの

**`refs`（および同格の参照束）に載せない** **方がよい**もの。

| 区分 | 理由の要約 |
|------|------------|
| **required 系を想起させるもの** | **`dimensions`/`measures` のキー列挙**、**「必須」語**に **つながる配列**。**§6** の **論理 requirement** に **接続**しやすい |
| **gating 解釈につながるもの** | **blocked / visible** 等の **単一ラベル**、**readiness**。**§8** |
| **候補の妥当性や recommendation を読ませるもの** | **013 のスコア**、**011 の `decision_recommendation`**、**候補の良否**。**011/013 の窓**は **別 GET** |
| **詳細 payload の埋め込み** | **`cells`・`evidence` 全文**・**review_points 全文**。**`GET .../read-artifact` 等**へ **委ねる** |
| **path 正本に見える識別子列挙** | **006 logical path 文字列の集合**。**canonical の列挙**と **誤読**されうる |
| **`metadata` 束の複製や上書き** | **`refs` 内の `metadata_id` 等**は **既に latest データセット由来**。**別の「正」**を **ここで** **宣言しない** |

### 14.4 一文定義案

**追加点**を **一文で** **言い切る** **案**（**1〜3**）。**実装・キー名は未定**（**§14.6**）。

1. **案1（推奨寄り）**: **「この table に対して latest の read-only 成果物（001 観測・002 判定）へ進むための ID が、他成果物と同様に空文字か UUID で分かる参照束である。」**

**本メモの推し**: **案1** を **主**とし、**案2** を **短い補足**として **併記**してよい。**案3** は **境界の宣言**として **強い**が、**「URL」**に **寄せすぎる**と **フロント実装**に **見える**ため **任意**。

## 完了メモ: Phase 2 完了（review-points top-level additive 拡張の一区切り）

### 要約

`GET /api/v1/metadata/{metadata_id}/review-points/` について、Phase 2 として top-level を段階的に additive 拡張し、ここで **一旦区切る** と整理する。

- `review_required`
- `dataset_id`
- `review_points_count`
- `table_id`

### この段階で閉じる理由

- endpoint の top-level に、**周辺参照に必要になりやすい最低限の情報**が揃ったとみなせる。
- 追加は **単純スカラー**や **`review_points` に対する素朴な派生（件数）**に留まっており、フェーズの輪郭がまだ読みやすい。
- 実装が **`AnalysisMetadataReviewPointsView` に局所化**されており、横断的な仕様書き換えを伴わずに収められている。
- **OpenAPI / migration / docs の変更なし**で進められたため、「契約の静かな拡張」としての Phase 2 の範囲に収まっている。

このうえ **意味づけを持たないまま** top-level をさらに増やし続けると、**何が観測で何が解釈か**の境界が曖昧になりやすい。よって **Phase 2 はここで一旦閉じる** 判断とする。

### 今回あえて入れていないもの

- **required / gating / recommendation / canonical** の正式化

### 位置づけ

## required 制度化の前に残す内部メモ

### 今回残す内部メモ

- **どの観点**なら **非生成候補**へ **寄せうるか**
- **どの観点**なら **caution** で **残しうるか**
- **skip 説明**を **将来** **どこで** **扱うか**

### required / canonical へまだ接続しない理由

- 上記を **すぐ** **`required_fields`** や **canonical** に **接続しない**（**文案4**）。**006 logical path** とも **つなげない**（**文案5**）。**candidate existence policy** の **判断根拠**は **`review_signal`** と **候補側条件**に **限定**して **扱う**。

### 次段実装へ渡す粒度

- **次段**に **実装**が **ある**としても、**まず** **内部ルール**の **追加**か、**小さな説明補助**に **留める** **粒度**と **する**。

**締め**: **制度化の前**に **残す** **設計メモ**として **本節群**を **置く**。

## 完了メモ: Phase 3 ここまでの一区切り（005 outcome → 013 参照面の最小接続と限定的な存在制御）

### 要約

**Phase 3** は、**005 outcome** を **013** が **どう読むか**を **最小参照面**として **固定**し、その **effect** を **内部要約**・**短い注意文**・**限定的な存在制御**に **留める**ことで、**非破壊**に **接続境界**を **整えた** **段**である（**文案1**）。**additive field** の **拡張**ではなく、**責務分離**・**内部要約**・**限定的な effect 接続**を **優先**した。**実装**は **`review_signal`** の **内部化**・**`risk_notes`** への **最小反映**・**blocking** 下の **限定的な stub 非生成**までに **留めた**。**public API**・**OpenAPI**・**migration** は **触れていない**。

### この段で固定できたこと

- **005 正本**／**013 consumer**／**011 補助**／**006 導線**の **線**を **維持**した。
- **013 evidence** と **006 logical path** を **接続しない**ことを **維持**した。
- **013** は **review** の **意味**や **state machine** を **所有しない**ことを **固定**した（**文案4**）。
- **`has_blocking_review_gap`** を **全面停止**の **スイッチ**に **しない**ことを **固定**した（**文案2**）。
- **blocking** を **強く効かせる候補**と **caution** で **残しうる候補**を **分けて扱う** **方針**を **固定**した。
- **非生成対象**の **拡張**は **category 羅列**ではなく、**判断観点先行**で **整理する** **方針**を **固定**した。

### 実装として入った最小差分

- **005 outcome** の **要約信号**として **`review_signal`** を **導入**した。
- **013** 側の **stub candidate** は **`review_signal`** **だけ**を **見る** **形**に **寄せた**。
- **`risk_notes`** に **review 由来**の **注意文**を **追加**した。
- **blocking** 下では **`summary_stub` かつ `metadata.time_axis` あり**の **場合のみ** **非生成候補**へ **寄せた**（**文案2**）。**それ以外**の **候補**は **shape 不変**の **まま残る**。
- **`suppression_applied`** の **既存投影**とは **統合していない**。
- **011**／**priority**／**confidence**／**required 制度**への **波及**は **まだ入れていない**。

### 今回あえて入れていないもの

- **public field** の **追加**
- **OpenAPI 本文**の **改稿**
- **migration**
- **`required_fields`** の **制度化**
- **canonical** の **導入**
- **candidate 全停止** **ルール**
- **skip 理由**の **public 化**
- **011** **複合判定**
- **logical path** との **接続**
- **candidate priority**／**confidence** の **制御**
- **`_audit_suppression_from_session`** との **統合**

いずれも **制度化**の **粒度**が **異なる**ため、**本段**では **あえて接続していない**（**文案3**）。

### 境界メモ

- **005** の **意味正本**を **013 側**へ **持ち込まない**。
- **013** は **consumer** として **review outcome** の **要約信号**だけを **使う**（**文案4**）。
- **effect** は **候補表示面**／**限定的な存在制御**に **留める**。
- **required**／**canonical**／**logical path** は **別整理**の **まま維持**する。
- **skip 説明面**は **将来** **必要**になり **うる**が、**現段**では **制度化しない**。

### 次フェーズへの引き継ぎ

- **次**に **進む** **場合**の **検討対象**は、おおむね **次の順**で **立ちうる**。

  1. **非生成対象**を **広げるかどうか**
  2. **skip 説明**を **どう扱うか**
  3. **required 制度**や **public 説明面**へ **接続するか**

- **いずれも** **まず docs で整理してから** が **前提**である（**文案5**）。**続ける余地**は **ある**が、**今** **ここで無理に広げない**。**現時点**では、Phase 3 の **目的**である **「最小接続と限定的 effect」** は **達成済み**として **一区切り**で **よい**（**本書**は **最終確定**の **代替**ではない）。
