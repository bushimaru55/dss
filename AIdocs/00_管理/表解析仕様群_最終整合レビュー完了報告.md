## 2. 目的

本書は、表解析コア仕様群および **SPEC-TI-014（API）** / **SPEC-TI-015（DB）** までを含め、**ここまで実施した整合確認の到達点**と、**次フェーズに送る論点**を一枚に整理する。目的は次のとおりである。

- 仕様フェーズで **どこまで合意が取れたか** を明文化する。
- **完了扱いにしてよい範囲**と、**実装・運用設計に回すべき範囲**を切り分ける。
- **揺り戻しを増やさない**ための「閉じ方」を共有する。

本書は **終了宣言**ではなく、**レビュー完了の記録と引き継ぎ**である。

## 4. 到達点サマリ

### 4.1 コア仕様群

- **観測〜判定〜正規化〜分析メタ〜人確認**の **責務分割**は、001〜005 および 011/013 の関係で **一貫した説明が可能**な状態にある。
- **標準経路**は **`001 → 002 → 003 → 004 → 005`** と整理できる。

### 4.2 SPEC-TI-006（データ契約）

- **DTO 骨格、主要 ID、列挙の分離**（例: **002 の `decision` と 011 の `decision_recommendation`**）が **文書上明示**されている。
- **完全 JSON Schema・全 DTO の機械可読パッケージ**は **Phase 4／次工程**の領域として位置づけられており、**本フェーズで無理に固定しない**判断が一貫している。

### 4.3 SPEC-TI-014 / 015（実装接続）

- **014** は **HTTP 詳細の最終表**までは含めずとも、**実行系／参照系の分離、識別子契約、ライフサイクル、エラー観点**まで **API 契約としての骨格**が揃った。
- **015** は **job 系と artifact 系の分離、世代／lineage、主要テーブル粒度**まで **DB 設計としての骨格**が揃った。
- **014 ↔ 015** について、**`evaluation_ref` ⇔ `evaluation_id`、`suggestion_run_ref` ⇔ `suggestion_run_id`** の **レイヤ差**が **両文書で補足済み**である。

### 4.4 「実装接続仕様フェーズ」への位置づけ

- **概念〜契約の地図**（006＋014＋015）は **揃った**。
- **叩き台のコード（OpenAPI、マイグレーション）** は **未着手でよい**段階として切り分け可能である。

## 5. 整合確認済み事項

### 5.1 責務境界（再掲）

| 仕様 | 正本とする内容 |
|------|----------------|
| 001 | 観測 |
| 002 | 判定（**`decision`**） |
| 003 | 正規化（**`dataset_id`** の主成果物） |
| 004 | 分析意味メタ（**`metadata_id`**、**`review_points`** の意味） |
| 005 | 人確認フロー（**`HumanReviewSession`・状態遷移・suppression**） |
| 006 | データ契約の骨格（DTO・主要 ref） |
| 011 | **risk signal / `decision_recommendation` / explanation** |
| 013 | 分析候補生成の実行（**主入力は 004／`metadata_id`**） |
| 014 | **API 契約** |
| 015 | **永続化設計** |

**経路・接続の前提（確認済み）**

- **005 の主入力**は **004 の `review_required` / `review_points`** に依存する、という整理で **矛盾が出ない**。
- **005 は suppression の正本**、**011 は recommendation 系の正本**、**013 は 004 を主入力とし 011/005 を読む側**、という整理で **014/015 の記述と整合**している。

### 5.2 用語・ID / ref

- **`session_id`** を **`HumanReviewSession` の正とする**方針で **014/015 が衝突しない**。
- **`evaluation_ref` / `suggestion_run_ref`（API）** と **`evaluation_id` / `suggestion_run_id`（DB 主キー）** の **対応**が **文書化済み**。
- **002 `decision` と 011 `decision_recommendation` の混同**が **API/DB 記述上、避けられている**。

### 5.3 014 ↔ 015

- **実行 API → `analysis_job` / `job_stage_run` 等**、**成果物参照 → 各 artifact テーブル**という **受け渡しの筋**が通る。
- **ジョブ表記**は **`analysis_job` / `job_stage_run`** に **揃え済み**（`job_run` 単独表記の整理）。

### 5.4 INDEX / メタデータ

- **INDEX** に **014/015 の版・パス**が **実ファイルと一致**している。
- **014/015** の **version 0.1、depends_on、相互リンク**は **整合チェックで問題なし**（詳細は整合チェック時点の記録に依る）。

以下は **本レビュー時点で「仕様フェーズの地図として十分」** とみなし、**同一論点を繰り返し掘り返さない**ことを推奨する。

| 事項 | 閉じる理由 |
|------|------------|
| **責務境界の骨格** | 001〜015 の役割分担が文書間で説明可能である。 |
| **006 の 0.5 時点のデータ契約** | Step 1〜3 の追補が入り、**014/015 の前提として使える**。 |
| **014/015 の 0.1 Draft** | **API 契約・DB 契約の章立てと識別子橋渡し**まで到達しており、**OpenAPI/DDL なしでも「何を実装するか」が議論できる**。 |
| **014↔015 の ref 対応補足** | **レイヤ差**が明示され、**実装時の迷いどころが減っている**。 |
| **INDEX の登録** | **発見可能性**が確保されている。 |

**今フェーズで固定しない（再オープンしない）ことの例**

- **HTTP ステータスと 012 コードの完全対応表**（**014 次版＋OpenAPI**）。
- **DDL 型・制約の最終形**（**015 次版＋リポジトリのマイグレーション**）。
- **RLS・マルチテナントの具体ポリシー**（**インフラ／アプリ設計**）。

## 8. 推奨する次工程

**推奨順（一例）**

1. **OpenAPI 叩き台**（主要リソース・主要エンドポイント・エラーシェイプ）。正本は **014＋006**。
2. **DDL 叩き台**（015 の論理テーブルを **1:1 で落とす**）。正本は **015**。
3. **対応表**（**API 操作 × 006 フィールド × DB 列**）。正本は **三者の突合**、格納は **リポジトリの docs またはスプレッドシート**でも可。
4. **実装前レビュー**（権限、冪等、stale、rerun の **受け入れ条件**）。

**この順の意図**: **契約を機械可読にしつつ**、**永続化と同じタイミングでズレを潰す**ため。
