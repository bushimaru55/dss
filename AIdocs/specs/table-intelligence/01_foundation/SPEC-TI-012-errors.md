## 文書目的

表解析パイプライン全体における**エラー・例外・失敗時の返し方**、**ログ／監査**、**再試行／人確認／候補抑制分岐**の扱いの**正本**を定義する。

**明記（責務の正本）**

- **001** は読取、**002** は判定、**003** は正規化、**004** は意味メタデータ、**005** は人確認、**011** は信頼度、**013** は候補生成の各**正本**であり、**012 は異常時の扱い（分類・致命性・遷移・メッセージ方針）の正本**である。  
- **012 は読取・判定・正規化・意味メタ生成・人確認・候補生成・信頼度計算を再実行しない**。各仕様が返す**状態・警告・失敗**を**分類**し、**ジョブ／API／UI／運用**が同じ語彙で扱えるようにする。  
- **012 は各仕様から返ってくる異常状態を分類し、「どう扱うか」**（続行・部分続行・中断・人確認・再試行）を定義する。  
- **利用者向け**には**過度に技術的なメッセージを出さない**。**運用向け**には**原因追跡可能**な情報（コード・ステージ・参照 ID）を残す。  
- **011** には**不確実性・失敗状態**を**特徴量化**できる材料が渡ること（数式は 011 正本）。  
- **013** には**候補抑制・安全側制御**の**前提**が渡ること（候補内容は 013 正本）。

## エラー処理の位置づけ

表解析パイプラインにおいて、012 は次に限定する。

1. **分類の正本**: 上流が出した**事実**を、**012 の分類軸**（種類・致命性）に載せる。  
2. **再実行禁止**: 012 は**パイプラインステージを呼び直さない**。  
3. **一貫した返却**: ジョブ・API が **同じ構造**で利用者／運用に返せる**ペイロード方針**を定める。  
4. **横断**: 単一ステージの「正しさ」は各 SPEC の正本とし、012 は**横断的な扱い**のみを定義する。

---

## エラー分類の原則

1. **事実と解釈の分離**: 001 の `parse_warnings` は**観測事実**が主。012 はそれを**致命性・遷移**に**写像**する。  
2. **ステージ語彙の尊重**: `NEEDS_REVIEW` は**002 の decision** 語彙。012 は**別の語で上書きせず**、「012 上の扱い＝人確認キュー投入可」など**関係**を定義する。  
3. **部分成功の明示**: `PARTIAL` は**003 の正規化**に近い語彙を**012 でも「部分続行」**として解釈しうる（定義は下表）。  
4. **コードの単一性**: 同一事象に**複数の理由コード**を付けうるが、**主 reason** を 1 つ推奨（監査のため）。

### 状態語彙の整理（012 横断）

| 語彙 | 主な出所 | 012 での位置づけ（暫定） |
|------|-----------|---------------------------|
| **`REJECT`** | 002 `JudgmentResult.decision` | **表候補として採用しない**。下流（003 以降）は**スキップまたは空成果**。ジョブは**継続しうる**（他表処理）。 |
| **`FAILED`** | 003 `normalization_status` 等 | **当該ステージで成果物が実質使えない**。**中断**または**部分ジョブ失敗**は致命性で決定。 |
| **`PARTIAL`** | 003 等 | **一部のみ利用可能**。005／011／013 へ**制約付き**で渡す。 |
| **`NEEDS_REVIEW`** | 002 `decision` | **自動確定しない**。012 は**人確認キュー**への**遷移を許可**する異常クラスとする。 |
| **`BLOCK`／候補抑制** | 005 `suggestion_suppression_level`、013 | **利用・提案のブロック／制限**。012 は**ジョブ致命**とは限らない。 |

## 異常の種類の整理

012 は少なくとも次の**種別**を区別する（相互に排他でない場合あり）。

| 種別 | 典型の出所（参照） |
|------|---------------------|
| **入力不正** | ファイル形式・サイズ・認可（014 前提） |
| **読取不能** | 001 実行時例外・I/O |
| **表未検出** | 001・0 件＋warnings |
| **見出し未確定** | 002／010 前提の曖昧さ → `NEEDS_REVIEW` |
| **判定競合** | 002 `evidence` の矛盾 |
| **正規化不能** | 003 `FAILED`・binding 不能 |
| **メタ生成不能** | 004 の失敗・空メタ |
| **人確認未解消** | 005 `UNRESOLVED` 等 |
| **候補生成抑制** | 005 suppression・013 空／抑制 |
| **内部処理例外** | 実装バグ・想定外例外 |

## 致命性の段階

| 段階 | 意味（012） | 例 |
|------|-------------|-----|
| **続行可能** | 警告のみ。下流へ**そのまま**渡す。 | 軽微な `parse_warnings` |
| **部分続行可能** | **一部表・一部成果物**のみ有効。 | 003 `PARTIAL` |
| **人確認で救済可能** | 自動では確定できないが**人入力後**に再開しうる。 | 002 `NEEDS_REVIEW` |
| **再試行可能** | **同一ジョブ**または**自動リトライ**で回復しうる。 | 一時 I/O |
| **中断すべき** | **これ以上の自動処理は危険または無意味**。 | 入力破綻、セキュリティ |

## 上流仕様との接続

| 接続 | 整理 |
|------|------|
| **001 `parse_warnings` と 012** | warnings は**観測事実の正本**（001）。012 は**ジョブ結果パッケージ**に載せる**理由コード・致命性**を**別層**で付与。 |
| **002 `NEEDS_REVIEW` と 012** | `NEEDS_REVIEW` は**判定の語彙**。012 は**「エラー」ではなくゲーティング**として扱い、**FATAL にしない**ことを推奨。 |
| **003 `PARTIAL`／`FAILED`** | `PARTIAL`＝**部分続行**、`FAILED`＝**当該表の下流スキップ**（012 のマッピング）。 |
| **004 `review_required`** | **メタ上の要確認**。012 は**人確認遷移**と**論理積**しうる。 |
| **005 unresolved／escalated** | **ワークフロー状態**。012 は**ジョブ完了条件**との**関係**のみを定義（詳細は 005）。 |
| **011 特徴量化** | **失敗・不確実性**の**件数・フラグ**を 011 へ。**012 は「渡す事実」の網羅**をここで示す。 |
| **013 suppression** | **候補の可否**。012 は**ジョブ結果**に**抑制の事実**を**載せる**方針（013 のアルゴリズムは変更しない）。 |

## ステージ別エラー一覧

| ステージ | 主な入力となる異常の形 | 012 での分類ラベル（例・暫定） |
|----------|-------------------------|--------------------------------|
| 読取 | `parse_warnings`、Artifact 欠損、ファイル形式 | `READ_*` |
| 判定 | `REJECT`／`NEEDS_REVIEW`、evidence 競合 | `JUDGE_*` |
| 正規化 | `PARTIAL`／`FAILED`、binding 欠落 | `NORM_*` |
| メタ生成 | `review_required`、生成不能 | `META_*` |
| 人確認 | 未解消、スキップ、タイムアウト（概念） | `HUMAN_*` |

## 読取段階のエラー

- **入力不正**: ファイル破損、許容外形式、サイズ超過 → **再試行不可**または**同一入力の再アップロード**。理由コード `TI_ERR_INPUT_*`（暫定）。  
- **読取不能**: ライブラリ例外、メモリ枯渇 → **再試行可能**（同ジョブ）を既定としうる。  
- **表未検出**: 001 の「0 件」＋ `parse_warnings`。**致命ではない**場合があり、**ジョブ継続**可。012 は**利用者向け**に「表が見つかりませんでした」等（文言は 007）。  
- **001 の `parse_warnings`**: **観測事実**の列挙。**012 のエラーイベント**とは別オブジェクト。012 は warnings を**集約し**、**致命性**と**理由コード**を**付与**しうる。

---

## 判定段階のエラー

- **見出し未確定**／**判定競合**: 002 の `NEEDS_REVIEW`＋`evidence`。012 は**人確認遷移**を**許可**するパスとする。  
- **`REJECT`**: 002 正本の語彙。012 は**下流スキップ**・**ジョブ部分成功**の**マッピング**を定義。  
- **`NEEDS_REVIEW` と 012 の「エラー」**: `NEEDS_REVIEW` は**ビジネス上の異常**ではなく**ゲーティング**。012 は**エラーレベル FATAL にしない**ことを推奨（暫定）。

---

## 正規化段階のエラー

- **`PARTIAL`**: **続行可能**な範囲で 004 へ渡す。012 は**警告レベル**＋**説明**を付与。  
- **`FAILED`**: **当該表の下流**は**スキップ**または**空**。ジョブ全体は**他表があれば継続**しうる。  
- **正規化不能**: binding 不能・入力契約破壊。`FAILED` に近い扱い。

---

## メタ生成段階のエラー

- **メタ生成不能**: 004 が**空メタ**または**エラー終了**した場合。012 は**理由コード**を `META_*` にマッピング。  
- **`review_required`**: **エラーではない**が、**012 の「人確認要」フラグ**と**論理積**しうる（005 連携）。

---

## 人確認段階のエラー

- **未解消**（`UNRESOLVED`）: 005 正本。012 は**ジョブ完了**を**ブロックしない**場合と**ブロックする**場合を**製品方針**で選ぶ（暫定: **ブロックしない**＋**フラグ**）。  
- **escalated**（概念）: 長時間未対応等。**運用向け**に**エスカレーションコード**を付与（詳細は未確定）。  
- **005 の unresolved と 012**: 012 は**ワークフロー状態**ではなく**パイプライン結果パッケージ**の**タグ**として扱う。

---

## 信頼度・候補生成への影響

- **011**: **失敗・PARTIAL・warnings 件数**等が**特徴量**として渡る（011 正本）。012 は**「渡すべき事実」**の**チェックリスト**をここで示す。  
- **013**: **`suggestion_suppression_level`**・**空候補**・**suppression_applied**。012 は**ジョブ結果**に**抑制理由**を**載せる**方針を定義（013 が生成した**内容**は変更しない）。

## 再試行方針

| 種別 | 再試行（暫定） |
|------|----------------|
| 一時的 I/O・タイムアウト | **自動再試行**しうる（回数上限は製品）。 |
| 入力データ同一の論理エラー | **再試行しない**（人確認または入力修正）。 |
| 002 `NEEDS_REVIEW` 解消後 | **再パイプライン**は 005／014 の正本（012 は**条件**のみ）。 |
| 内部処理例外（バグ疑い） | **中断**＋**運用向け**にスタック要約（PII なし）。 |

---

## 人確認遷移方針

- **自動再試行すべきでない**が**救済可能** → **人確認**（002 `NEEDS_REVIEW`、004 `review_required`、003 `PARTIAL` で利用者判断が有効なもの）。  
- **即中断すべき** → **入力破綻**・**セキュリティ**・**データ損失リスク**（暫定リストは Phase4）。  
- **012 は 005 の画面フローを定義しない**。**「キューに載せる条件」**の**参照元**を示す。

---

## 利用者向け表示方針

- **伝えるもの**: **何ができなかったか（業務言語）**、**次に何をすべきか**（再アップロード、別ファイル、担当者へ連絡等）。  
- **見せないもの**: **内部スタックトレース**、**生の reason コード**（サポートモード除く）。  
- **trace／code／cause**: **利用者**には **code は原則非表示**。**サポート画面**では **code ＋短い cause** を許容（007）。

---

## 運用ログ・監査方針

- **残すもの**: **理由コード**、**ステージ ID**、**table_id／job_id**、**相関 ID**、**時刻**、**主要 upstream 参照**（artifact_id 等）。  
- **保存先・保持期間**: **未確定**（015／運用規程）。  
- **PII**: ログに**セル値を載せない**原則（必要時は**マスク**方針を 014／015）。

---

## エラーコード／理由コード方針

- **プレフィックス**: `TI_ERR_`（エラー）／`TI_WARN_`（警告）等で**ステージを区別**しうる（暫定）。  
- **001 `parse_warnings.code` との関係**: **マッピング表**を 012／006 で**同期**（Phase4）。**同一事象で二重定義**を避ける。  
- **HTTP との対応**: **014 正本**。012 は**論理コード**のみを正本とする。
