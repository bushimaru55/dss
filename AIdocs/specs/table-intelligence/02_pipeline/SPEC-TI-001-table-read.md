## 文書目的

ファイルおよびシートから、**表として後続処理へ渡す候補領域**を安全に抽出し、セル格子・結合・一次メタを **機械可読な `TableReadArtifact`** として固定する。本仕様は表解析コアの**入口**であり、**意味の最終確定**や**最終分類**は行わない。

---

## スコープ

- 対応する入力形式と読取単位（ファイル／シート）
- **表候補**の最小定義と、候補領域の検出方針
- タイトル・注記・装飾・空白と**表本体候補**の一次切り分け（候補・ラベル付けまで）
- セル属性の**取得対象**（何を取るか）。値の解釈の最終確定は含めない
- 空白／結合／装飾セルの**一次扱い**
- 複数表混在シートでの**1 表候補あたりの範囲**の扱い
- `TableReadArtifact` の構造（006 の必須契約に準拠）と**再現性**要件
- 表未検出・読取不能時の返却と `parse_warnings` の使い方
- 人確認（005）へ渡すための**保持すべき情報**のうち、読取段階で決めうる部分

## 用語定義

| 用語 | 説明 |
|------|------|
| 表候補（TableCandidate） | 1 つの矩形領域として後続へ渡す「表らしき塊」。`table_id` と `bbox` で識別（[SPEC-TI-006](../01_foundation/SPEC-TI-006-io-data.md)）。 |
| 表本体候補（data region hint） | 表候補内で、データセルが存在すると推定される矩形。読取段階では**推定**であり、最終確定は 002／010。 |
| タイトル候補／注記候補 | 表候補の上下左右などに付随するテキスト領域の**候補**。ラベルと矩形（またはセル集合）を付与する。 |
| TableReadArtifact | 1 表候補に対応する読取結果の成果物。006 で必須フィールドが定義される。 |
| 一次構造化 | セルを座標付きで列挙し、結合・型・書式など**観測可能な属性**を載せること。意味解釈は含めない。 |
| 再現性 | 同一入力バイト列・同一 `read_profile`・同一エンジンバージョン下で、同一 `TableReadArtifact`（または定義済みの同値関係）が得られること。 |

## 表読取の位置づけ

表解析パイプラインにおいて、表読取は次の役割に限定する。

1. **候補抽出**: シート上のどの矩形を `TableCandidate` とするか決める。  
2. **一次構造化**: その矩形内のセル・結合・観測可能属性を `TableReadArtifact` に格納する。  
3. **不確実性の明示**: 推定・欠落・警告を `parse_warnings`（および暫定の拡張メタ）で後続へ引き渡す。

**明記**: 表読取は**意味の最終確定**ではなく、**候補抽出と一次構造化**までである。

---

## 対応対象

### 入力形式（初版）

| 種別 | 初版での位置づけ |
|------|------------------|
| Excel（`.xlsx` 等） | **対象（主）**。シート単位で読む。 |
| CSV | **対象**。1 ファイルを 1 シート相当として扱う（行／列は 0 始まりインデックスで表現）。 |
| その他（PDF 画像化表など） | **未確定**または対象外。暫定: OCR／画像パイプラインは別仕様。 |

### バージョン・実装差分

同一形式でもライブラリ差で微妙に異なる場合があるため、**`read_profile`**（後述）と**エンジン識別子**を Artifact に含め、再現性の前提とする。

---

## 原本保全方針

- **入力バイト列のハッシュ**（例: SHA-256）を `File` レベルで保持する（006・014 の範囲。読取仕様では「保持が必要」旨のみ明記）。  
- `TableReadArtifact` 内では、**原本セルへの参照**（アドレスまたは `(row,column)`）を必ず残し、後続の trace の基礎とする。  
- 読取は**元ファイルを改変しない**。派生物（キャッシュ展開など）がある場合は `read_profile` または `parse_warnings` に記録する。

---

## 読取単位

- **主単位**: **1 シート**（Excel）。CSV は 1 ファイル＝1 シート。  
- **1 シート内**に複数の `TableCandidate` を返しうる（複数表混在）。  
- `TableCandidate` は [SPEC-TI-006](../01_foundation/SPEC-TI-006-io-data.md) に従い `sheet_id`・`bbox`・`read_artifact_ref` を持つ。

## 座標系と bbox（Draft 0.2 暫定標準）

本節は **001 時点の暫定標準**である。将来 [SPEC-TI-006](../01_foundation/SPEC-TI-006-io-data.md)・[SPEC-TI-010](../01_foundation/SPEC-TI-010-heading-model.md)・[SPEC-TI-003](SPEC-TI-003-normalization.md) と**用語・範囲解釈を完全に一本化**する対象とし、差分が出た場合は **006 を正**として各仕様を MINOR で揃える。

### セル座標

- **行・列インデックスは 0 始まり**とする（上端行＝0、左端列＝0）。

### bbox（外接矩形）

- **`bbox` は 0-based の inclusive 範囲**を標準とする。  
- 形式は **`(row_min, col_min, row_max, col_max)`** とし、次を意味する。  
  - **`row_min`, `col_min`**: 領域に含まれる**最小**の行／列インデックス。  
  - **`row_max`, `col_max`**: 領域に含まれる**最大**の行／列インデックス（**両端とも領域に含まれる**）。  
- **1 行×1 列**の場合は `row_min = row_max` かつ `col_min = col_max` でよい。

### A1 等の表記

- **A1 形式は人間向けの補助表現**であり、機械可読な正本ではない。表示・ログ用に付けてよいが、**契約上の正は 0-based の `(row, col)` および上記 `bbox`** とする。

## 表候補の最小定義（中心論点 1）

### 何をもって「表候補」とみなすか

**暫定標準（Draft 0.2）**: シート上の**軸に沿った矩形領域**であって、次の **既定条件**または**例外条件**のいずれかを満たすものを 1 つの表候補としてよい。

#### 既定条件（原則: 2×2）

次をすべて満たす場合、**表候補とする（推奨の既定）**。

1. **最小サイズ**: データとして解釈しうるセルが、**少なくとも 2 行 2 列**分存在する（結合セルは結合後の占有範囲で数える）。  
2. **格子の痕跡**: 連続する非空セルまたは罫線／書式による「行・列の並び」の**痕跡**がある。完全な罫線は必須としない。  
3. **孤立ブロック**: 明らかに離れた別ブロックは**別の表候補**とする（ブロック間の距離閾値は実装パラメータとし、`read_profile` で固定）。

#### 例外条件（1×N / N×1 の緩和）

実務上の**小規模表・帳票断片**の取りこぼしを減らすため、**2×2 を満たさない**場合でも、次をすべて満たすとき **表候補として扱ってよい（例外許容）**。  
**最終的に「表として採用するか」は SPEC-TI-002 に委譲**する（読取は候補の抽出まで）。

1. **一方向の系列性**: **1×N（縦）**または **N×1（横）**のいずれかで、**複数セル**が連続しており、かつ**同種の値パターン**（例: 数値＋ラベル列、日付列、キー列）が並ぶ**痕跡**がある。  
2. **表／欄の意図の観測**: 罫線・背景帯・罫り表格子の片側・「項目／値」様の並び等、**表または欄の一部**と読める視覚的・配置的手掛かりのいずれかがある（手掛かりが弱い場合は `parse_warnings` に観測事実として残す）。  
3. **ノイズでないことの否定が困難でもよいが、明らかにノイズなら除外**: 下記「表候補としない代表例」に該当する場合は**候補にしない**。

**例外として許容しうる候補の例**（例示。確定分類ではない）:

- **単一列の縦型明細**（項目名＋値が縦に並ぶ、連番＋金額のみの列など）  
- **単一行の横型小表**（期間や指標が横一列に並ぶミニ表）  
- **帳票断片**（本表の一角のみが取り込まれたブロック）  
- **凡例・補助表に近いブロック**（記号対応表、単位・区分の小表）

**表候補としない代表例**（読取段階で**原則除外**。迷う場合は `parse_warnings` に記録）:

- **単なるセクション見出し**（1 行の大見出しのみで格子が続かない）  
- **単発の注記文**（長文テキスト 1 セルまたは連続するが表格子でない説明文）  
- **単一セルのラベルのみ**（周囲に系列が広がらない）  

### 最低限必要な構成要素

各 `TableCandidate` について、少なくとも次を確定させる。

| 要素 | 説明 |
|------|------|
| `table_id` | 006 に従う一意 ID。 |
| `bbox` | 上記「座標系と bbox（Draft 0.2 暫定標準）」に従う外接矩形。 |
| `TableReadArtifact` | 006 必須: `artifact_id`, `table_id`, `cells`, `merges`, `parse_warnings`。 |

### タイトル・注記・装飾・空白と表本体の切り分け

- **表本体候補**: `bbox` 内のうち、データ格子が最も密な矩形を**デフォルトの data region hint** とする（アルゴリズムは実装詳細。結果は `parse_warnings` または拡張メタで説明可能にする）。  
- **タイトル候補**: 通常は表本体の**上側隣接**の連続セル（1 行または結合）を候補とする。確定はしない。  
- **注記候補**: 表本体の下側・右側の小さなテキストブロックを候補とする。  
- **装飾・空白**: 表候補の `bbox` には含めつつ、`cells` は**稀疏標準**に従い、**`cells` にキーが無いセルは空**とみなす（後述）。注記候補への寄せは任意。  

典型例・拾わない例・曖昧時の扱いの詳細は、後述「**タイトル・注記・表本体の切り分け**」を参照する。

### 複数表混在時にどこまでを 1 つの表候補とするか

- **原則**: 視覚的・格子として**明確に分離**されたブロックは**別 table_id**。  
- **曖昧な場合**: 1 つにまとめると誤結合リスクが高いときは**分割優先**。分割しすぎた場合は 002／005 で統合・修正を想定。  
- **009 との整合**: 009 が想定する「複数表混在シート」は、読取では**複数 `TableCandidate`** として表現する（シート単位 taxonomy は持たない）。

## 表候補領域の検出方針

### 検出の考え方

**暫定**: 次の 2 段階を推奨する（実装は任意だが、外部仕様としての出力は同等であること）。

1. **シード探索**: 非空セル・数値セル・罫線付きセル等をシードとし、近傍に同様の格子が広がる領域を拡張する。  
2. **矩形化**: 拡張結果を軸平行の外接矩形 `bbox` に正規化する。

### 安定性のためのルール

- **決定性**: 同一シード集合に対し、拡張順序に依存しないよう、**セル走査順（上→下、左→右）**を標準とする。  
- **閾値**: 空行・空列の「幅」が何セルまで同一表とみなすかは `read_profile` に記載する。  
- **009 参照**: 009 の「読取（001）重点」は、検出時の**オプションフラグ**（例: 小計行を含む広い bbox を許容するか）として解釈し、**taxonomy 確定は行わない**。

## タイトル・注記・表本体の切り分け

| 区分 | 読取での扱い |
|------|----------------|
| 表本体候補 | `bbox` 内の主格子。**`cells` は稀疏標準**（後述）。`bbox` 内で `cells` に現れない位置は**空セル**とみなす。 |
| タイトル候補 | `region_hints.title`（任意）に矩形またはセル範囲を列挙。テキストの「意味」は確定しない。 |
| 注記候補 | `region_hints.notes[]`（任意）に同様。 |
| 未使用領域 | `bbox` に含めるが、002 が「表外」として扱えるよう、原則は空セル扱いまたは注記候補へ。 |

**タイトル候補として拾う典型例**（候補であって確定ではない）:

- 表本体の**直上**に接する**連続セル行**（1 行のタイトル文、複製列見出しの上の帯）  
- **結合セル**で横断している「見出し風」の 1 行（書式・フォントが本文と異なる場合は `parse_warnings` または書式ヒントで残す）

**注記候補として拾う典型例**:

- 表本体の**直下**の但し書き・脚注ブロック（複数行の小さな矩形）  
- 表本体の**右側**の補足欄（単位説明、注釈、参照先）

**候補として拾わない／拾わないことを優先する例**:

- 表から**離れた**章見出し・大見出し（同一シートの別セクションタイトル）  
- **ページ共通ヘッダ／フッタ**に相当する繰り返しテキスト（帳票の固定文言。検出できる場合は表候補の `region_hints` に含めず、シートレベル扱いを検討／**未確定**）  

**曖昧な場合**: 読取は**確定しない**。`parse_warnings` に観測事実（例: `scope: table`, `message` に「タイトル候補と章見出しの区別が困難」等）を載せ、002／005 へ渡す。

**任意拡張フィールド（006 同期予定）**: `TableReadArtifact` に `region_hints` を追加する場合、006 の MINOR 更新で正式化する。Draft 0.2 では **JSON 内のオプションキー**として定義し、無い実装は `parse_warnings` のみで代替可能。

## セル属性の取得対象

各セル（または結合アンカー）について、次を**取得対象**とする。

| 属性 | 必須／任意 | 説明 |
|------|------------|------|
| 座標 | 必須 | 0 始まり `(row, column)`。結合はアンカー基準を `merges` と整合。 |
| 表示値（raw display） | 必須 | ユーザーが画面上で見える文字列に近い値。 |
| データ型（library-native） | 推奨 | 数値・日付・文字列等。解釈の最終確定は 002／003。 |
| 数式 | 任意 | 数式文字列。評価値は **キャッシュ結果**を採用（未計算は warning）。 |
| 書式ヒント | 任意 | 太字・背景色・罫線有無等。010 の疑似見出し検出の入力になりうる。 |

## 空白セルの扱い

- **標準プロファイル（Draft 0.2 暫定）**: `cells` は**稀疏表現**とする。  
- **`bbox` 内で `cells` にキーが存在しないセル位置は、すべて「空セル」とみなす**（＝値は空、結合アンカーでない限りデータセルとして列挙しない）。  
- **全セル展開**（`bbox` 内の全座標を明示的に列挙する表現）は**標準ではない**（後述「`cells` の標準表現」）。  
- **parse_warnings**: 広範囲空白による bbox 肥大化が疑わしい場合、または「空が多すぎて表本体の推定が不安定」な場合に付与。  
- **最終的な欠損意味**（意図的空白か不明か）は 002／005。

---

## 結合セルの扱い

- **merges**: [SPEC-TI-006](../01_foundation/SPEC-TI-006-io-data.md) 必須。各結合は `{ anchor_row, anchor_col, height, width }` 形式（暫定）で列挙。  
- **cells（標準プロファイル）**: **アンカーセルに値を集約**し、**被覆セルは `cells` に重複保持しない**（省略＝稀疏）。同一値の繰り返しで被覆セルを埋めない。  
- **例外／将来拡張プロファイル**: 同一値を被覆セルすべてに複製する**全展開風**の表現が必要な場合は、`read_profile` で **`cells_representation: full_materialized`** のように**明示したプロファイル**でのみ許容する（標準プロファイルとの混在禁止）。  
- **010 への接続**: 010 はアンカーをノード代表とするため、001 の merges と矛盾があってはならない。

---

## 装飾セルの扱い

- 装飾のみで値が空のセルは、**標準プロファイル（稀疏）**では、**書式ヒント等の観測がある場合のみ** `cells` に**アンカー**として載せる（値は空文字または null 等、006 Phase4 で型固定）。**観測が無い**装飾のみのセルは **`cells` に省略**し、**空セル**とみなす。  
- 「見出しらしさ」は 002／010 の領域。読取は**特徴を落とさない**ことが目的。

---

## 複数表混在シートの扱い

- 検出された各ブロックごとに **`TableCandidate` を分離**する。  
- **順序**: `table_index`（任意）をシート内の上→下、左→右で付与することを推奨。  
- **重複 bbox**: 許可しない。重複候補が出た場合は警告し、ルールで一方を採用（`read_profile` 明示）。

**分割を強く支持する条件**（読取段階の**分割判断根拠**。最終的な統合・再解釈は 002／005 に委譲）:

- **`read_profile` で定義した幅以上**の**完全空白行**または**完全空白列**がブロック間にある  
- **見出し様式**（書式・フォント・背景・罫線パターン）がブロックごとに**明確に切り替わる**  
- **単位表記**（通貨、単位、％表記のルール等）がブロックごとに**一貫して異なる**  
- **表線・背景パターン**が**明確に断絶**している（罫線の箱が別、帯色が切れる等）

上記のいずれかに該当し、かつ誤結合リスクが高い場合は**別 `table_id` に分割する**ことを優先する。判断が曖昧なときは分割し、`parse_warnings` に観測事実を残す。

## TableReadArtifact の定義

### SPEC-TI-006 との整合（必須）

[SPEC-TI-006 §5.4](../01_foundation/SPEC-TI-006-io-data.md) の必須フィールドを**変更せず**満たす。

| フィールド | 本書での意味 |
|------------|----------------|
| `artifact_id` | この読取結果の ID。 |
| `table_id` | 対応する `TableCandidate.table_id`。 |
| `cells` | セル集合。**標準プロファイルでは稀疏表現**（後述）。キーは `"R{row}C{col}"` 形式（暫定）または配列。内部は 006 Phase4 で JSON Schema 固定。 |
| `merges` | 結合一覧。 |
| `parse_warnings[]` | 読取段階の警告・観測事実（後述の暫定最小構造）。 |

### `cells` の標準表現（Draft 0.2 暫定標準）

- **標準プロファイル**: **`cells` は稀疏表現**とする。  
- **`bbox`（inclusive）内で `cells` にキーが無い位置は、すべて空セル**とみなす。  
- **結合セル**は **`merges` で表現**し、**被覆セルは原則 `cells` に重複保持しない**（値はアンカーに集約）。  
- **全セル展開表現**（`bbox` 内の全座標を明示的に列挙）: **標準ではない**。**例外的・将来拡張**として、`read_profile` で **`cells_representation: full_materialized`** 等を**明示したプロファイル**でのみ許容する。標準プロファイルとの混在は不可。  

本節は **001 の責務範囲**で読取結果の**標準的な保持形**を固定するものであり、006 の**必須フィールド名は変更しない**。

### 推奨するオプションキー（初版・006 MINOR で同期予定）

相互運用のため、次を**任意**で許容する。

| キー | 説明 |
|------|------|
| `coordinate_system` | 例: `{ "cell_index": "0-based", "bbox": "0-based-inclusive" }`（本書の暫定標準と一致させる）。 |
| `read_profile_id` | 閾値・**cells 表現方針（sparse / full_materialized）**・結合の一貫性等の識別子。 |
| `engine` | `{ "name", "version" }` |
| `region_hints` | `title` / `notes[]` / `data_region` の矩形（いずれも **0-based inclusive** 推奨）。 |
| `stable_fingerprint` | 正規化済みセル集合のハッシュ（再現性検証用・任意）。 |

## TableReadArtifact の再現性要件（中心論点 2）

### 同一入力から同一出力を得るための前提

次をそろえたとき、**ビット同一**または仕様で定義する**同値**として扱う。

1. 同一の**入力バイト列**（ファイル）。  
2. 同一の **`read_profile_id`**（閾値・**cells の稀疏／全展開**・結合の一貫性・分割閾値を含む）。  
3. 同一の **`engine.name` / `engine.version`**。  
4. 同一の **タイムゾーン・ロケール**（日付セル解釈に影響する場合）。未使用なら明示的に固定。

### 読取処理で許容する裁量の範囲

**許容**: `read_profile` で宣言されたパラメータの範囲内の差。  
**不許容**: 同一 `read_profile`・同一エンジンバージョンで、セルキー集合や merge 集合が無秩序に変わること。

### 表候補抽出結果の安定性の担保

- 走査順・ソートキーを仕様化する。  
- 同値セル（空・同一表示値）が並ぶ場合のタイブレークを定義する（例: 行優先）。  
- 乱数を用いない。ヒューリスティックのタイブレークは文書化する。

### 後続へ渡す最小データ構造の固定

- **最小**: 006 の 5 フィールド。  
- **010 向け**: `merges` と各セルの座標・表示値・（あれば）書式ヒント。  
- **002 向け**: `parse_warnings` に「観測事実」を構造化して載せる（スコアは載せない）。

#### `parse_warnings` 暫定最小構造（Draft 0.2）

`parse_warnings` の各要素は、**観測事実の構造化**を目的とし、**スコア・最終判断（要確認確定等）は含めない**（002／011／005 の領域）。

**少なくとも次を推奨／暫定標準とする**（012 へのコードマッピングは 012 側で確定）。

| フィールド | 必須 | 説明 |
|------------|------|------|
| `code` | ✓ | 012 へ接続可能な文字列（例: `TI_READ_*` 接頭辞を想定）。 |
| `message` | ✓ | 人間可読の説明（内部用語のままでよい。UI 文言は 007）。 |
| `scope` | ✓ | `file` \| `sheet` \| `table` \| `cell` のいずれか。 |
| `target_range` または `cell` | 推奨 | 対象範囲。**`target_range`** は `{ row_min, col_min, row_max, col_max }`（**0-based inclusive**）。**`cell`** は `{ row, col }`。 |
| `severity` | 任意 | `info` \| `warning` \| `error` 等（**暫定**。012 の重大度と突合予定）。 |

同一事象に複数解釈がある場合は、`message` に並記せず、**要素を分けて**載せるか、`code` を分ける。

## 補足メモ（初版の外枠）

### Draft 0.2 時点で未確定の論点

- **010 の `source_range` 表記**（A1 文字列か座標ペアか）と **001 の bbox** の**相互変換規約**の全文固定（現状は 001 側で 0-based inclusive を暫定標準化済み）。  
- **ページ共通ヘッダ／フッタ**をシートレベルでどう扱うか（表候補外に出すか、専用 artifact か）。  
- 巨大ファイル・メモリ上限とストリーミング読取の要否。  
- 数式未計算セルの本番ポリシー（Excel）。  
- `parse_warnings.severity` と **012 の重大度**の対応表。  

### SPEC-TI-002 に引き継ぐ事項

- 表候補が**一覧／クロス／帳票**のどれに近いかの**確定**と `taxonomy_code` 割当。  
- **1×N／N×1 例外**で抽出された候補を、**表として採用するか／別タイプに寄せるか**の最終判断。  
- タイトル／注記／表本体の**最終ラベル**と見出しセル候補の**採否**。  
- **`parse_warnings`（`scope`／`target_range`／`cell` 付き）**を根拠とした **JudgmentResult.evidence[]** の組み立て。  
- **分割根拠**（空白帯・様式切替・単位差異等）の**解釈確定**（統合・再分割の要否）。  

### SPEC-TI-003 に引き継ぐ事項

- 確定した見出し・行種別に基づく**縦持ち／ワイド化**と `trace_map`。  
- 型・単位・集計行の**意味付き正規化**。  
- **`bbox` およびセル座標が 0-based inclusive** である前提の下での**原本参照**（010／006 との表記同期は別途）。  

### 既存 009 / 010 / 006 との自己点検結果（Draft 0.2）

| 観点 | 結果 |
|------|------|
| 006 `TableReadArtifact` 必須フィールド | **整合**。名称・必須性を変更していない。`parse_warnings` の内部構造は 001 の暫定標準であり 006 の「配列必須」と矛盾しない。 |
| 009 | **衝突なし**。読取は taxonomy 確定を行わず、マトリクスはオプション解釈に留めた。1×N 例外は**候補抽出**の緩和であり分類確定ではない。 |
| 010 | **整合意図**。001 は bbox／セルを 0-based inclusive で固定。010 の `source_range` 表記は**別仕様で同期予定**（レビュー整理メモの論点と一致）。 |
| 越境 | 分類・見出し確定・正規化詳細・スコア式は**委譲**として明示。 |
