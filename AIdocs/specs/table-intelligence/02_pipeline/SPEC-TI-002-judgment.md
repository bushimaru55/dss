## 文書目的

[SPEC-TI-001](SPEC-TI-001-table-read.md) の `TableReadArtifact`（および対応する `TableCandidate`）を入力とし、**表種別**・**見出し候補の採否**・**行種別**・**列（属性／指標／時系列）の役割**・**単位候補**・**1×N／N×1 例外候補の採否**・**人確認要否の一次判断**を、**根拠つき**で確定する規約を定義する。

**明記**: [SPEC-TI-001](SPEC-TI-001-table-read.md) の出力は**候補と一次構造化**に留まる。**採否・分類・一次判断**は本仕様（002）が担う。

---

## スコープ

- 判定の入力・出力（`JudgmentResult`）と、判定の**原則・優先順位**
- 表種別（`taxonomy_code`）の**適用**（語彙・境界は 009 正本）
- 見出し候補の**採否**と、010 の `HeadingTree` に載せる**前提となる割当**（木のデータモデル自体は 010）
- 明細行／集計行／注記行の**一次分類**
- 属性列／指標列／時系列列の**一次判定**（最終の grain・measure 定義は 004／003 側）
- 単位行・単位セル候補の**検出と採否**
- 001 が許容した **1×N／N×1 例外候補**の**表としての採否**
- 曖昧性の**保持**（無理な確定をしない）と `NEEDS_REVIEW` への接続
- `JudgmentResult.decision`・`evidence[]` の**構造方針**
- `parse_warnings`（001 由来および判定で追加する観測）の**参照・接続方針**

## 判定ロジックの位置づけ

表解析パイプラインにおいて、002 は次を行う。

1. **解釈**: 001 の候補を、009 の語彙で**どの表種別として読むか**を決める（確定できなければ `TI_TABLE_UNKNOWN` 等）。  
2. **構造化の前提**: 010 に従い、**どのセルを見出しとして採用するか**を決める。  
3. **格子の意味づけ**: 行・列の**役割の一次ラベル**（明細／集計／注記、属性／指標／時系列候補、単位候補）を付与する。  
4. **ゲーティング**: 曖昧性が残る場合は**無理に確定せず**、`NEEDS_REVIEW` と evidence を残す（011／005 と接続）。

**明記**:

- **009** は表分類の**語彙と境界の正本**であり、002 はそれを**適用して判定**する。  
- **010** は見出し**モデル**の正本であり、002 は**「この候補をどう採用するか」**の判定に徹する。  
- **003** は正規化の正本であり、002 は**変換そのものを行わない**。

## 判定の原則

1. **証跡必須**: 主要な結論はすべて **evidence** に紐づける。  
2. **正本尊重**: `taxonomy_code` は **009 の enum のみ**。002 は新語を増やさない。  
3. **無理な確定禁止**: 境界・役割が割り切れない場合は **`NEEDS_REVIEW`** とし、**複数仮説**を evidence に残す。  
4. **001 尊重**: 001 の `parse_warnings` は**観測事実**として信頼し、**矛盾する断定**をしない。必要は evidence で `parse_warnings` を参照。  
5. **011 非侵襲**: 本書は**信頼度の数式を固定しない**。`decision` は**ルールベース一次結論**とし、011 が合成して変えうる余地を残す（暫定）。  
6. **優先順位**: 矛盾するルールがある場合は **ルール優先順位表**（後述）に従う。

### ルール優先順位（暫定）

| 優先度（高い順） | 内容 |
|------------------|------|
| P0 | **入力欠損・破綻**（Artifact 不正）→ `REJECT` または 012 連携。 |
| P1 | **009 上の型の必須軸欠落**（例: CROSSTAB で行軸・列軸の一方が成立しない）→ `NEEDS_REVIEW` または `UNKNOWN`。 |
| P2 | **見出し採否**（010 整合） |
| P3 | **行種別・列役割** |
| P4 | **単位候補** |

## 表種別判定

### 方針

- **語彙・定義**は [SPEC-TI-009 §6](../01_foundation/SPEC-TI-009-table-taxonomy.md) に従う。  
- 002 は、001 の格子・値分布・見出し候補・warnings を手掛かりに **1 つの `taxonomy_code`（single-label 既定）** を割り当てる。  
- **確信が持てない**場合は `TI_TABLE_UNKNOWN` とし、`NEEDS_REVIEW` を検討する。

### taxonomy の単一ラベル既定と副候補の保持（Draft 0.2）

- **`JudgmentResult.taxonomy_code`（006 正本）は常に単一値**とする。009 の **single-label 既定**に合わせ、002 は**主たる1分類**だけをこのフィールドに載せる。  
- **複数仮説**がある場合、**副候補**は `evidence[].details` に保持してよい（暫定キー例: `alternate_taxonomies[]`）。各要素に `code`（009 enum）・`rationale`（短い根拠）・`targets`（任意）を含める。  
- 副候補は **011／005** の判断材料とし、`taxonomy_code` を配列化しない。

### 初版での観点（例示。詳細ルールは別表で拡張）

- **LIST_DETAIL**: 行方向にレコードが並び、列が属性／指標として一貫している。  
- **CROSSTAB**: 行軸・列軸の交差に**度量**が載る。  
- **TIME_SERIES**: 時系列が**横持ち列**で繰り返す。  
- **KEY_VALUE**: キー列＋値列の**縦並び**が支配的。  
- **FORM_REPORT**: 格子が弱く、ブロック分割・注記が支配的（001 の region_hints／warnings と整合）。  
- **PIVOT_LIKE**: 小計・階層、折り畳み痕跡。  
- **LOOKUP_MATRIX**: 記号・カテゴリが主で度量が従属または不在。

**境界**（009 と重複しないよう**参照のみ**）: LIST_DETAIL と PIVOT_LIKE、FORM_REPORT と LIST_DETAIL は 009 §8 に従い、002 は**断定できなければ UNKNOWN／要確認**。

## 見出し候補判定

### 方針

- **見出しになりうるセル**を 001 の書式・位置・テキストから抽出（001 の範囲）。002 は**採否**する。  
- **データ領域**との境界が曖昧な場合、**両方の仮説**を evidence に残す。  
- 採用結果は [SPEC-TI-010](../01_foundation/SPEC-TI-010-heading-model.md) の `HeadingTree` 構築へ引き渡す（**010 が木構造の正本**）。

### 採否の観点（暫定）

- タイトル行・多段見出し・結合見出し（001 `merges` と整合）  
- **疑似見出し**（010 定義）: 書式のみ・空に近いテキスト → **保留**または `NEEDS_REVIEW`  
- 009 のタイプごとの**期待パターン**（例: CROSSTAB は行／列見出しの双方）との整合

### 見出し帯とデータ開始行の境界（Draft 0.2）

- **見出し帯**: 連続する見出し候補行（単段／多段）で、**データセル（値の列）がまだ始まっていない**行の集合とみなす。  
- **データ開始行**: 見出し帯の直下（または右方）で、**一様なデータ型・値パターン**が列方向に伸び始める最初の行。  
- **判断**: 境界は **配置手掛かり**（空行・罫線・背景帯の切れ目）と **型手掛かり**（数値列の出現）を組み合わせる。**曖昧**なら帯を広めに取り、`NEEDS_REVIEW`＋evidence に両仮説の行範囲を載せる。

### 多段結合見出しの上位／下位（Draft 0.2）

- **上位（外側）**: 行方向では**より上**、列方向では**より左**の結合見出し。010 の `level` が小さい側に相当。  
- **下位（内側）**: データ領域に近い側の結合見出し。  
- **002 の役割**: `merges` とセル値から **親子の順序**を決め、010 のノード親子に**矛盾しない**割当を evidence（`details.heading_levels` 等）に残す。結合が **T 字・L 字**で親子が割れない場合は `NEEDS_REVIEW`。

### 疑似見出しの採用／保留／不採用（Draft 0.2）

| 扱い | 条件（暫定） |
|------|----------------|
| **採用** | 周辺に**対になる見出しパターン**があり、データ領域との位置関係が一貫。書式が見出し帯と連続。 |
| **保留** | 書式のみ・キャプション空に近いが、**軸が成立しうる**。`decision` は `NEEDS_REVIEW` を検討。 |
| **不採用** | データセルと区別がつかず、**ノイズ**と判断。注記・装飾に落とす根拠を evidence に残す。 |

### タイトル候補の昇格／降格（Draft 0.2）

- **昇格（見出し／表題として採用強化）**: 001 `region_hints.title` または表本体直上の結合行で、**表の外縁と整列**し、**データ列と独立**している。全文が短く表題的。  
- **降格（見出し帯へ合流 or 注記扱い）**: 表題としては広すぎる・**章見出し**に近い位置、または**但し書き**様の文体。  
- **曖昧**なら昇格も降格も確定せず、`details.title_hypotheses` と `NEEDS_REVIEW` を推奨。

## 行種別判定

### 対象ラベル（暫定 enum・006 同期予定）

- `ROW_DETAIL`（明細）  
- `ROW_SUBTOTAL` / `ROW_GRAND_TOTAL`（集計。細分は 003）  
- `ROW_NOTE`（注記・但し書き）  
- `ROW_HEADER_BAND`（見出し帯。行ではあるがデータ行ではない）  
- `ROW_UNKNOWN`

### 方針

- キーワード（「合計」「小計」「計」）、書式、位置、数式パターンを**手掛かり**とする。  
- **集計行をデータから除外するか**は 003 と合意（本書では**タグ付け**まで）。曖昧なら `NEEDS_REVIEW`。

### 行種別の衝突時優先原則（Draft 0.2）

複数ルールが異なる行種別を指す場合、次の**高い方を優先**する（同順位なら `ROW_UNKNOWN`＋`NEEDS_REVIEW`）。

1. **文言手掛かり**（「合計」「小計」「計」「Subtotal」等の強い語）  
2. **配置手掛かり**（表下端・外側の「合計」行、罫線で切られた帯）  
3. **型手掛かり**（数式・参照・集計関数の痕跡）  
4. **周辺構造手掛かり**（上に見出し帯、下に注記ブロックと連続）

## 列種別判定

### 対象ラベル（暫定）

- `COL_ATTRIBUTE`（属性／ディメンション候補）  
- `COL_MEASURE`（指標／メジャー候補）  
- `COL_TIME`（時系列軸候補）  
- `COL_UNIT`（単位列候補。単位行と重複しうる）  
- `COL_NOTE`  
- `COL_UNKNOWN`

### 方針

- 型（数値・日付・文字列）、列ヘッダ文言、列方向の単調性（時系列らしさ）を用いる。  
- **最終的な measure／dimension 確定**は 004。002 は**候補と根拠**まで。

### 列役割の衝突時優先原則（Draft 0.2）

複数ルールが異なる列役割を指す場合、次の**高い方を優先**する（同順位なら `COL_UNKNOWN`＋`NEEDS_REVIEW`）。**手掛かりの種類の優先順は行種別と同一**（文言 → 配置 → 型 → 周辺構造）とする。

1. **文言手掛かり**（「合計」「単位」「年月」「金額」等のヘッダ語）  
2. **配置手掛かり**（左端の属性列、右端の注記列、日付が横に並ぶ帯）  
3. **型手掛かり**（列全体が数値／日付／文字列のどれに寄るか）  
4. **周辺構造手掛かり**（ヘッダ帯との連続、同一ブロック内での列の役割の一貫性）

## 単位候補判定

### 対象

- **単位行**（表上部・下部の「単位:」行）  
- **単位セル**（ヘッダ近傍の `(円)` 等）  
- **単位列**（`COL_UNIT` と併記されうる）

### 適用範囲の仮ラベル（Draft 0.2）

単位候補は evidence の `details` 等で、次のいずれか（複数可）としてタグ付けする。

| 仮ラベル | 意味 |
|----------|------|
| `UNIT_SCOPE_TABLE` | **表全体**に効く単位（表題横、先頭行の「単位:」等）。 |
| `UNIT_SCOPE_COLUMN` | **特定列**に紐づく単位（列ヘッダ内の `(千円)` 等）。 |
| `UNIT_SCOPE_CELLS` | **特定セル群**（離れた注記セル、凡例内の単位ブロック）。 |

### 方針

- 正規表現・キーワード・括弧パターンは**暫定**。一致しても**確定**せず、evidence に**マッチ根拠**を残す。  
- 複数単位が混在する場合は `NEEDS_REVIEW`。

## 1×N／N×1 例外候補の扱い

[SPEC-TI-001](SPEC-TI-001-table-read.md) は **2×2 未満でも表候補**を許容しうる。002 は次を判定する。

### 採否の観点

| 観点 | 説明 |
|------|------|
| 系列性 | 縦／横に**複数セル**の系列があり、**ラベル＋値**等のパターンがあるか。 |
| ノイズ分離 | **単発注記・章見出し・単一ラベル**（001 が除外しきれなかった場合）でないか。 |
| 009 への載せ方 | KEY_VALUE・LIST_DETAIL・LOOKUP_MATRIX・UNKNOWN のどれが最も近いか。 |

### 結論の扱い

- **表として採用**できると判断 → 適切な `taxonomy_code` を割当、`AUTO_ACCEPT` 候補。  
- **表ではない**→ **`REJECT`**（下記「`REJECT` と `UNKNOWN`＋`NEEDS_REVIEW` の境界」）。**表としては成立しうるが分類不能**なら `TI_TABLE_UNKNOWN`＋`NEEDS_REVIEW`。  
- いずれも **evidence に 001 例外該当の事実**と**採否理由**を残す。

---

## 曖昧性の扱い

- **複数の taxonomy** が同点に見える、見出し帯が二重に読める、集計行と明細が判然としない等は**曖昧**とみなす。  
- 対応:  
  - `decision = NEEDS_REVIEW` を**強く推奨**、または `taxonomy_code=UNKNOWN`。  
  - evidence に **`ambiguity` ブロック**（仮説列挙・各仮説の短い根拠）を載せる。  
- **無理に AUTO_ACCEPT しない**。

## 人確認要否判定

### `REJECT` と `TI_TABLE_UNKNOWN`＋`NEEDS_REVIEW` の境界（Draft 0.2 暫定標準）

| 結論 | 条件 |
|------|------|
| **`REJECT`** | **入力破綻**（`TableReadArtifact` 必須欠損・矛盾）、**表対象外**（ノイズのみ・001 が誤抽出した明らかな非表）、**表として成立しない**（格子・系列のいずれも説明不能で補正も困難）。 |
| **`taxonomy_code = TI_TABLE_UNKNOWN` かつ `NEEDS_REVIEW`** | **表としては成立しうる**（セル系列・手掛かりがある）が、**種別・軸・見出しの確定が困難**。複数仮説が残る。 |

**011 との関係**: 011 は**スコアと閾値**で `decision` を**再解釈**しうる。002 は**ルールベースの一次 `decision`**を出し、evidence を 011 に渡せる形にする（数式は 011）。

### `decision` の一次基準（暫定）

| decision | 条件（例） |
|----------|------------|
| `AUTO_ACCEPT` | 009 型の必須軸が満たされ、見出し・行／列役割の矛盾が少なく、001 warnings に致命的なものがない。 |
| `NEEDS_REVIEW` | 曖昧性・必須軸欠落・1×N 採否の疑義・単位競合・FORM_REPORT 相当のレイアウト不確実。上記境界に従い **`REJECT` に落とさない**ケース。 |
| `REJECT` | 上記境界表の **`REJECT`** に該当。 |

## evidence の構造方針

### 目的

- 判定の**再実行・監査・人確認・信頼度計算**に耐えること。  
- [SPEC-TI-001](SPEC-TI-001-table-read.md) の `parse_warnings` と**接続**できること。

### 各要素の推奨フィールド（暫定・006 の `evidence[]` の各オブジェクトの中身）

| フィールド | 必須 | 説明 |
|------------|------|------|
| `rule_id` | ✓ | 例: `J2-TAX-003`。 |
| `conclusion` | ✓ | そのルールが出した結論の要約（例: `taxonomy_code=TI_TABLE_LIST_DETAIL`）。 |
| `targets` | 推奨 | セル／範囲の配列。**0-based inclusive**（001 に準拠）。 |
| `refs_parse_warnings` | 任意 | 001 の `parse_warnings` の **index** または **code** の配列。 |
| `details` | 任意 | 行／列マップ、仮説列挙、`ambiguity` 等の構造化 JSON。 |
| `confidence_hint` | 任意 | **[SPEC-TI-011](SPEC-TI-011-confidence-scoring.md) への定性的ヒント**（例: `high`/`medium`/`low`）。**最終スコアではない**。011 が合成する前段の**ルール側の感触**のみを表す。**数式・重み・閾値は書かない**。 |
| `details.confidence_drivers` | 任意 | 011 合成の材料として、**減点・加点要因の短い列挙**を `details` 内に保持してよい（暫定。例: `{ "drivers": ["merge_ambiguous", "header_boundary_unclear"] }`）。 |

**禁止**: スコアの**最終確定値**、UI 文言、正規化後の列名の確定。

## parse_warnings の利用方針

- **出所**: 主に **001 の `TableReadArtifact.parse_warnings`**。002 は**原則として Artifact を破壊しない**。  
- **参照**: evidence の `refs_parse_warnings` で**観測事実にリンク**する。  
- **002 の追記**: 判定時に新たな観測があれば、**実装選択**として (a) 同一 Artifact の配列へ追記、または (b) evidence のみに閉じる。**006 で保存形式が確定するまで (b) を推奨**。  
- **スコア・最終判断**は `parse_warnings` に**書かない**（011／005）。

## 補足メモ（初版の外枠）

### この初版で未確定の論点

- `REJECT` の**運用**（ジョブ全体失敗 vs スキップ）と HTTP／012 の対応（本文では境界のみ暫定標準化済み）。  
- **011 による `decision` 上書き**の正式アルゴリズム順序。  
- 行／列マップを **006 に昇格**するか、別 artifact にするか。  
- ルール ID の**完全なレジストリ**（012 との対応）。

### SPEC-TI-003 に引き継ぐ事項

- 行種別・列役割・単位候補・確定 `taxonomy_code` に基づく**変換経路選択**。  
- 集計行を**データから切り離すか**、メタとして残すかの**最終方針**。  
- **0-based inclusive** 座標系に基づく `trace_map` 設計。

**002 → 003 の接続境界（要約）**

- **002 が 003 に渡すもの**: `evidence[]` の **`J2-ROW-001` / `J2-COL-001`**（`details.by_row_index` / `by_column_index`）。backend の materialize ではこれを **`dataset_payload.normalization_input_hints`** に**畳んだ形**で 003 が参照しうる。**候補ヒントであり、行種・列役割・taxonomy の意味確定ではない**（§行／列一次ラベル（J2-ROW / J2-COL スパイク）および [SPEC-TI-003](SPEC-TI-003-normalization.md) §MVP 実装接続）。  
- **003 がそのまま用いてよいもの**: **行／列 index ベース**のヒント、001 の **`raw_display` 転記**、**`column_slots` の参照面**（列 index と `cN` の対照）。  
- **003 が 002 由来だけで確定しないもの**: **taxonomy の最終意味**、[SPEC-TI-004](../03_analysis_human/SPEC-TI-004-analysis-metadata.md) における **dimension／measure**、**業務意味**、**merges／多段見出しの解決**。  
- **逃がし先**: **不一致・曖昧さを 003 が抱え込まず**、`trace_map`・**`NEEDS_REVIEW`・人確認**・上位仕様へ**委譲**する（002 は根拠を `evidence` に残し、003 は**再判定しない**）。

### SPEC-TI-005 に引き継ぐ事項

- `NEEDS_REVIEW` のときの**曖昧性タグ**・**複数仮説**（evidence.details）。  
- 001 `parse_warnings` との**突合**（利用者に見せる要約は 007）。  
- **UNKNOWN** からの**再判定**入力の前提。

### SPEC-TI-011 に引き継ぐ事項

- `confidence_hint` および evidence の**構造化された根拠**。  
- `AUTO_ACCEPT`／`NEEDS_REVIEW` の**閾値**設計。  
- 001 warnings・002 ルール違反の**減点要因**へのマッピング。

### SPEC-TI-001 / 009 / 010 / 006 との自己点検結果

| 観点 | 結果 |
|------|------|
| 001 | **整合**。候補・一次構造化に留め、002 が採否・分類を担当。Artifact 必須フィールドは変更しない方針。 |

## MVP 実装スパイク（backend 整合・2026-04）

本節は **002 全ルールの完成**ではなく、`JudgmentResult` を **根拠付き**に引き上げるための **最小境界**をロックする。

### P0 で `REJECT` に寄せる条件（最小）

| ID | 条件 |
|----|------|
| J2-P0-001 | `cells` が空（稀疏表に観測セルが 1 つも無い） |
| J2-P0-002 | `TableScope` の bbox が反転（`row_max < row_min` または `col_max < col_min`） |
| J2-P0-003 | いずれかのセル座標が bbox 外（0-based inclusive で検査） |
| J2-P0-004 | `parse_warnings` に **`severity: error`**、または code が `TI_READ_NO_TABLE_CANDIDATE` / `TI_READ_ARTIFACT_INVALID` / `TI_READ_BBOX_INVALID` のいずれか |

P0 発火時は **`taxonomy_code = TI_TABLE_UNKNOWN`**（006）、`decision = REJECT`。012／HTTP の運用対応は別途。

### evidence 最小形（スパイク）

各要素は少なくとも次を含む: **`rule_id`**, **`conclusion`**, **`targets`**（矩形またはセル参照・空配列可）, **`refs_parse_warnings`**（001 の **index** 列）, **`details`**。

- **`details.judge_profile_id`**: 実装バージョン識別子（例: `ti.judge.spike.v1`）。006 の `judge_profile_id` トップレベル化は **MINOR 見送り**でよい。
- **`refs_parse_warnings`**: `parse_warnings` が空なら `[]`。非空なら **全要素を index で参照**する `J2-WARN-001` を付与してよい。

### 薄い J2-TAX（スパイク）

P0 非該当かつ `REJECT` でない場合、観測特徴（行列スパン、数値／日付らしさ、先頭行の文字比率等）から **単一の `taxonomy_code`** を推定。自信が薄いときは **`TI_TABLE_UNKNOWN`** とし、**`details.alternate_taxonomies[]`** に副候補を閉じる（002 Draft 0.2 と整合）。

**v1（`ti.judge.spike.v1`）**: [SPEC-TI-009](../01_foundation/SPEC-TI-009-table-taxonomy.md) 各型の定義から **機械判定可能な最小の「軸」**（例: クロス表の行軸・列軸ラベル数、交差部の数値、時系列の列上日付、キー値の横幅、帳票の結合／充填率、ピボット様式キーワード）を **`details.taxonomy_009.axes_observed`** に記録し、仮分類に対して **軸ゲート**を適用する。ゲート不通過時は **`TI_TABLE_UNKNOWN`** へ落とすか、代替候補のうちゲート通過の型へ **昇格**してよい（実装方針）。

### 行／列一次ラベル（J2-ROW / J2-COL スパイク）

P0 非該当時、**本文 §行種別／列種別**の暫定 enum を、001 の **稀疏セル＋bbox** だけから **候補**として付与する。`evidence[]` に **`J2-ROW-001`**（`details.by_row_index`）と **`J2-COL-001`**（`details.by_column_index`）を載せる。キーは **0-based 行／列インデックスの文字列**。**`details.primary_labels_schema`**（例: `ti.judgment.primary_labels.v1`）でスキーマを識別する。MVP 実装では 003 への誤確定を避けるため、値に **`ROW_NOTE_CANDIDATE`**、**`COL_ATTRIBUTE_CANDIDATE`**、**`COL_MEASURE_CANDIDATE`** 等の接尾辞を付けうる（§本文 enum との対応は実装コメントで明示）。

- **003 向け**: 正規化の経路選択・行の除外候補の**ヒント**として消費しうる。**確定の論理行・`trace_map`・grain は 003 正本**。
- **010 非侵襲**: `HeadingTree` や `cell_bindings` は作らない。見出しの**採否確定**は 010／後続 002 の責務。
- **004 非侵襲**: `dimensions`／`measures` の意味確定には**しない**（`details.intent` で「semantic lock-in でない」と明示してよい）。
